## 1. プロジェクト概要

### 1.1 背景

慢性疾患（ME/CFS、線維筋痛症、Long COVID、うつ病ほか 65 種類）の患者向け健康日記 SPA を、Firebase 中心構成から **GCP コンテナ + PostgreSQL** を中核とするサーバサイド主体アーキテクチャへ全面刷新する。

### 1.2 リアーキテクチャの動機

旧仕様（ブラウザ vanilla JS SPA + Firebase）には以下の構造的限界がある:

| 動機カテゴリ | 内容 |
|---|---|
| **データ主権**（最優先） | 個人の医療情報を Firestore（Google 配下）に置いており、将来の規制対応・コンプライアンス・データ統制の自由度が低い |
| データモデル | Firestore のコレクション構造ではリレーショナルクエリ・集計・JOIN が困難 |
| パフォーマンス制御 | Firestore の挙動は外部からチューニング不可、コスト直撃 |
| コード品質 | vanilla JS + 13 モジュール構成の保守限界 |
| **SSO 要件** | ValuationMatrix / zen-track 等の自社サービス群との共通アカウント |

詳細: [ADR-0001](../adr/0001-rearchitecture-from-firebase-to-gcp.md)

### 1.3 目的・ゴール

- 既存ユーザー（数十人）の体験を **同じドメイン `cares.advisers.jp` で継続**しながら、内部を全面刷新する
- データを **自社管理 DB（PostgreSQL）に保管**し、データ主権を確立
- **SSO Lv2 基盤**（Keycloak 中央 IdP）を整備、cares / ValuationMatrix / zen-track での共通アカウント運用
- ブラウザに PII を残さない設計で **共有 PC・ブラウザ拡張機能・マルウェアの passive read 攻撃** から防護
- 機能面はおおむね旧仕様を継続移植、本格刷新は MVP 後

### 1.4 KPI / 成功指標

| 指標 | 定義 |
|---|---|
| MAU | 月次アクティブ利用者 |
| 1 ユーザあたり月間記録日数 | 利用継続の指標 |
| AI 解析実行数 / ユーザ | エンゲージメント |
| 医師提出レポート生成数 | アウトカム貢献 |
| ユーザ当たり AI コスト | コストキャップ運用の妥当性 |
| 移行成功率（旧 → 新） | カットオーバーの成否（数十人ベース） |

### 1.5 解決したい課題（旧仕様と同等、医療面の価値提案は継続）

| 課題 | 影響 | 本サービスの対応 |
|---|---|---|
| 多次元データの記録負荷が高い | 継続できない | 9 カテゴリの最低限フォーム + Plaud 連携 |
| 医師との診察時間が短く伝え切れない | 治療方針に反映されない | 医師提出用レポート自動生成 |
| 慢性疾患の最新研究にアクセスしづらい | 治療選択肢を知らないまま | PubMed 検索 + 日本語要約 |
| 自分の症状が他人にどれくらいいるか分からない | 孤立感 | 疾患別疫学データを AI 応答に注入 |
| 専門医を見つけられない | 受診遅延 | 専門家リスト + メール送信 |
| **個人の医療データを SaaS 事業者に預けている不安** | 将来規制・データ流出リスク | **GCP 自社管理 DB + ブラウザ PII 禁止 + 監査ログユーザー可視化** |

## 2. スコープ

### 2.1 MVP 対象範囲

- **認証**: Keycloak セルフホスト + OIDC RP
  - Google ログイン / Email+Password / Magic Link / 匿名（SSO 圏外）
  - 2FA はユーザー任意
- **疾患選択**: 65 種類 + カスタム（旧仕様を継続）
- **データ記録**: 9 カテゴリ（症状・バイタル・薬・睡眠・活動・食事・血液検査・気分・写真）
- **AI 解析**: 旧仕様の主要機能を継続
  - 日々の解析 / 深堀解析（1 日 1 回）/ 会話型相談 / タイムライン解析 / 画像解析
- **論文検索**: PubMed 連携 + AI 日本語要約
- **医師提出レポート**: カルテ風レポート生成
- **3 つの受け手向けサマリー**: 医師 / SNS / 近親者
- **専門家連携**:
  - 専門家メール送信
  - テレヘルス / カウンセラー / 緊急相談窓口リスト
- **外部連携**:
  - Google Calendar / ICS
  - Plaud（音声録音メール取り込み）
  - CSV エクスポート（**MVP 必須**）
- **管理者機能**:
  - プロンプト管理、API キー設定、Firebase 設定（→ Secret Manager 設定）、データ管理
  - 三段フェイルセーフ認可（Keycloak ロール + オーナーメール + ブレークグラス）
- **データポータビリティ**: JSON / CSV / PDF / FHIR エクスポート
- **監査ログ**: ユーザー本人にも公開（誰がいつアクセスしたか）
- **データ削除ポリシー**: ソフト削除 → 30 日後ハード削除
- **多言語**: 10 言語対応（旧仕様継続）

### 2.2 フェーズ 5 後半（MVP 後）

- Fitbit 連携
- Apple Health 連携
- CSV インポート
- 家族・医師との限定共有（招待・範囲限定・期間限定）

### 2.3 対象外

- 医療機関向け SaaS / EHR 連携
- 処方・診断機能（教育的情報提供に限定）
- 患者同士のコミュニティ機能
- 動画記録
- ネイティブモバイルアプリ（PWA で代替）
- リアルタイム Push 通知
- B2B 契約モデル（スキーマでは tenant_id を持つが、MVP では未実装）

### 2.4 前提・制約条件

- 利用者は Web ブラウザ（モバイル含む）からアクセス
- 対応ブラウザ: モダンブラウザ最新 2 世代（Chrome / Edge / Safari / Firefox）
- 個人情報保護法（日本）に準拠
- データ保管リージョン: asia-northeast1（東京）
- AI 利用には API キーが Secret Manager に設定済みであること
- ブラウザに PII（健康データテキスト・写真）を永続化しない（IndexedDB は暗号化キャッシュとして許容）
- PWA オフラインモードはアプリシェルのみ。ユーザーデータはオンライン必須

## 3. ステークホルダー / ユーザー

### 3.1 ステークホルダー一覧

| ステークホルダー | 関心事 | 意思決定権 |
|---|---|---|
| プロダクトオーナー（運営） | 価値提供・運用継続性・データ主権 | スコープ・優先度 |
| 開発者 | 設計・実装・運用 | 技術選定 |
| 一般利用者（患者） | 使いやすさ・プライバシー | 利用継続 |
| 管理者（`agewaller@gmail.com`） | プロンプト・API 鍵管理・運用判断 | 運用判断 |
| 医療従事者 | 患者からの情報提供品質 | （間接的） |
| 自社他サービス（ValuationMatrix / zen-track） | SSO 連携 | アカウント連動仕様 |

### 3.2 利用者の役割

| 役割 | 説明 | アクセス可能機能 |
|---|---|---|
| 匿名（Anonymous） | Keycloak で匿名認証。SSO 圏外 | 限定機能（試用）、データは自分のセッションのみ |
| 一般利用者（User） | Keycloak で正規認証済み | 全 9 カテゴリ記録、AI 解析、論文検索、外部連携、エクスポート、SSO 連携 |
| 管理者（Admin） | Keycloak ロール `admin` または OWNER_EMAIL 一致 | 全機能 + 管理パネル（`/admin`） |

### 3.3 ペルソナ

**「平岡 みち子」 65 歳・主婦**（旧仕様継続）

- 線維筋痛症と高血圧で 3 つの病院を受診中
- スマートフォン（iPhone）の操作はメッセンジャーと写真程度
- 紙のお薬手帳と日記帳で記録しているが、医師に渡しても見てもらえない
- 「AI」「アプリ」というラベルに警戒感あり。専門用語より平易な日本語、ボタンが大きいことを重視

> ユーザ向け UI からは「AI」のリテラル語を出さない方針（旧仕様継続）

### 3.4 主要利用シナリオ

| ID | シナリオ | アクター | 旧仕様との差分 |
|---|---|---|---|
| US-01 | 初回登録（Google ログイン）→ 疾患選択 → 初回データ入力 | 一般利用者 | Firebase Auth → Keycloak |

| US-06 | 新しい治療の論文を PubMed で検索 → 日本語要約 | 一般利用者 | 同等 |

### 4.2 ビジネスルール

| ルール ID | ルール | 適用箇所 |
|---|---|---|
| BR-01 | 「本格的な深堀解析」は同日 1 回まで（JST 0 時で reset） | FN-AI-02 |
| BR-02 | AI プロンプトは共通システムプロンプトを先頭注入 + prompt caching 対象 | FN-AI-* |
| BR-03 | 14 軸処方（PRESCRIPTION_AXES）は `dayOfYear % 14` で全ユーザ同期 | FN-AI-08 |
| BR-04 | API 鍵は Secret Manager のみに保管。ブラウザに送らない・受け取らない | FN-AI-*、FN-ADMIN-02 |
| BR-05 | Claude モデル ID は MODEL_MAP 経由のエイリアス解決のみ（datestamped ID 禁止） | FN-AI-09 |
| BR-06 | アカウント削除はソフト削除 → 30 日後にハード削除 | FN-AUTH-07 |
| BR-07 | 管理者判定は **Keycloak ロール OR OWNER_EMAIL OR ブレークグラス** の三段 | FN-ADMIN-* |
| BR-08 | ユーザ入力を DOM に挿入する前に escape | 全画面 |
| BR-09 | ユーザ向け UI に「AI」リテラルを出さない | 全画面 |
| BR-10 | localStorage / sessionStorage に PII を書かない。IndexedDB は AES-GCM 暗号化のみ | 全機能 |
| BR-11 | データテーブルは全て `tenant_id` を持つ（MVP では tenant = user 個人） | スキーマ全体 |
| BR-12 | AI 呼び出しは全件 `ai_usage_logs` に記録、ユーザー単位コストキャップに反映 | FN-AI-* |
| BR-13 | 全データアクセス（read / write / delete / export）は `audit_logs` に記録 | 全機能 |
| BR-14 | 匿名ユーザーは SSO 圏外、サービス間遷移不可 | FN-AUTH-03 |
| BR-15 | フォームドラフトはサーバ側 `drafts` テーブルに autosave | FN-DIARY-06 |
| BR-16 | PII を含む HTML / JSON は `Cache-Control: no-store`、Service Worker でキャッシュ不可 | 全画面 |
| BR-17 | 画像データも PII。Signed URL で都度フェッチ、メモリのみ、IndexedDB にキャッシュしない | 写真表示 |
| BR-18 | 監査ログ表示は PII。表示中メモリのみ、IndexedDB にキャッシュしない | FN-AUDIT-01 |
| BR-19 | ログアウトは Backchannel logout で全サービス同時。「このサービスだけログアウト」は提供しない | FN-AUTH-06 |
| BR-20 | AI コストキャップ デフォルト月次 1000 円、上限到達で 422 拒否（縮退フォールバックなし） | FN-AI-10 |
| BR-21 | アカウント削除と個別レコード削除はとも 30 日のソフト削除猶予 | FN-AUTH-07, FN-DIARY-05 |
| BR-22 | 匿名ユーザーのデータも無期限保持。SSO 圏外だが cares 単独では制限なく利用可能 | FN-AUTH-03 |

## 5. 非機能要件

### 5.1 性能（暫定目標）

| 項目 | 目標 |
|---|---|
| 非 AI API レイテンシ | p95 < 500ms |
| AI 応答ストリーミング開始 | < 2 秒（以降は LLM 依存） |
| ページロード（LCP） | < 2.5 秒 |
| 日々の AI 解析（Sonnet 4.6） | 10〜20 秒 |
| 本格的な深堀解析（Opus 系） | 30〜60 秒 |
| 高速モード（Haiku 4.5） | 3〜8 秒 |
| 想定ストレージ | ユーザー 1 人あたり 100MB / 月（写真込み） |

### 5.2 可用性

- **SLA**: 明示的 SLA なし、ベストエフォート（MVP 数十人規模）
- 東京リージョン内の冗長性で対応
- マルチリージョン DR は MVP スコープ外
- AI プロバイダ障害時は **管理者の手動切替**（自動フォールバックは行わない）

### 5.3 セキュリティ

- 通信は全て HTTPS（TLS 1.2 以上）、HSTS 有効
- API 鍵は **Secret Manager のみに存在**、ブラウザ・DB には載せない
- 保管時暗号化: Cloud SQL / Cloud Storage の Google 管理鍵で自動暗号化（MVP は CMEK 不要）
- **ブラウザストレージ**:
  - localStorage / sessionStorage: PII 完全禁止
  - IndexedDB: AES-GCM 暗号化、セッションバウンド鍵、セッション中のみ TTL
  - Cookie: HttpOnly + Secure + SameSite=Strict のセッションのみ
- 認可ガード: 全 API で OIDC ID token 検証。管理 API は三段フェイルセーフ
- Origin 許可リスト: BE は cares の正規 Origin のみ受付

### 5.4 プライバシー / データ主権

- 全ユーザーデータは Cloud SQL に `tenant_id` 単位で永続化
- 1 件単位での編集・削除がすべての種別で可能
- ソフト削除 → 30 日後ハード削除
- いつでも全データを **JSON / CSV / PDF / FHIR** でエクスポート可能
- 監査ログをユーザー本人にも公開
- ユーザ向け UI ではサービス名（Anthropic 等）を出さない
- 個人特定情報は SNS 用要約から自動マスキング

### 5.5 ユーザビリティ / アクセシビリティ

- モバイルファースト（65 歳ペルソナが iPhone 主体）
- shadcn/ui の Radix UI ベースで a11y を確保
- ライトテーマ固定（高齢者の可読性配慮、ダークテーマは追加しない）
- `confirm()` / `alert()` は使用禁止（モバイルでブロック）。インライン UI で代替
- ボタン・テキストは大きめ、フォント・コントラスト・タップ領域を設計初期から要件化
- ユーザ向け表示は日本語、必要に応じて 10 言語切替

### 5.6 法令・コンプライアンス

- 個人情報保護法（日本）に準拠
- 同意取得 UI、開示請求対応、削除請求対応を MVP に実装
- 医療情報を扱うが **教育的情報提供** の範囲に限定。診断・処方は行わない
- GDPR / HIPAA 相当は将来必要時に対応（スキーマ設計で先行配慮）

### 5.7 移行・カットオーバー

- 既存ユーザー（数十人規模）はデータ・アカウントとも自動移行
- Email + Password ユーザーは初回ログインで強制パスワードリセット
- 匿名ユーザーは個別対応
- 旧システムは `legacy.cares.advisers.jp` として 90 日間読み取り可能で残す

## 6. 識別子の体系

旧仕様 [`../../docs_restruct/README.md`](../../docs_restruct/README.md) を踏襲。新仕様追加は `*-NEW`、廃止は `*-DEPRECATED`。

- 機能 ID: `FN-{ドメイン}-{連番}`
- 画面 ID: `SC-{連番}`（フェーズ 3 で画面一覧を作成）
- API: `{METHOD} {path}`（API 一覧 [`../detail-design/api/api-list.md`](../detail-design/api/api-list.md) で管理）
- ADR: `ADR-{NNNN}`（[索引](../adr/README.md)）
- テーブル: スネークケース複数形（例: `symptoms`, `ai_usage_logs`）

## 7. 用語集

| 用語 | 説明 |
|---|---|
| SSO Lv2 | 中央 IdP + フェデレーションによる真の SSO。サービス横断のユーザー ID 共通化 |
| Keycloak | OSS の中央 IdP。本仕様で採用 |
| global_uid | Keycloak が OIDC `sub` クレームで発行する、サービス横断のグローバル ID |
| user_link | 各サービスの `users` テーブルと `global_uid` を紐付けるテーブル |
| tenant_id | マルチテナント対応のためのテーブル列。MVP では個人ユーザー単位 |
| MODEL_MAP | CONFIG 上のモデル ID を AI Provider API ID に解決する辞書 |
| PROMPT_HEADER | 全 AI プロンプトに先頭注入される共通指示 |
| 14 軸処方 | 日替わりでローテーションする 14 種類のセルフケア軸 |
| Plaud | ユーザが利用する音声録音デバイス |
| break-glass トークン | 通常運用が壊れた時の緊急用管理者認証トークン |
| OWNER_EMAIL | オーナーのメールアドレス（環境変数）。Keycloak ロールに関係なく強制的に admin 扱い |
| ブラウザ PII 禁止 | localStorage / sessionStorage に PII を書かない原則。IndexedDB は暗号化のみ |
| セッションバウンド鍵 | ログイン後にサーバから払い出され、メモリと IndexedDB メタに保持される一時暗号鍵 |
| ソフト削除 | 削除フラグ付与のみ。30 日後にハード削除（物理削除）される |
| 監査ログ可視化 | 自分のデータに誰がいつアクセスしたかをユーザー本人に見せる仕組み |
| Backchannel logout | 1 つのサービスでログアウトしたら他サービスにも伝搬する SSO 機能 |
