## 1. 認証・認可の詳細設計

### 1.1 OIDC 認証フロー

詳細シーケンス: [`../basic-design/architecture/sequence-auth.mmd`](../basic-design/architecture/sequence-auth.mmd)

実装ポイント:

- Next.js の `auth.ts` で OIDC クライアント（`openid-client` パッケージ）を初期化
- セッションは Iron Session 等で **HttpOnly + Secure + SameSite=Strict Cookie** に格納
- セッション内容: `globalUid`, `userId`, `roles`, `expiresAt`
- セッション TTL: 24 時間、リフレッシュトークンで延長可能
- BE は `Authorization: Bearer <ID token>` または `Cookie: session=...` から検証

### 1.2 認可 Middleware

すべての BE エンドポイントを覆う共通 middleware:

```
要求受信
  ↓
1. session/token 抽出
  ↓
2. Keycloak JWKS で署名検証（キャッシュ済み）
  ↓
3. claim から global_uid 取得
  ↓
4. DB の user_link → users.id 解決
  ↓
5. ctx.user に格納
  ↓
6. 管理 API なら追加で 1.3 の三段チェック
  ↓
7. 監査ログ middleware が action を記録
  ↓
ハンドラ実行
```

### 1.3 管理者認可の三段フェイルセーフ

ADR-0011 で確定。実装は以下の OR 判定:

```typescript
function isAdmin(ctx: Context): boolean {
  const session = ctx.get('session')

  // 段 1: Keycloak ロール
  if (session.roles?.includes('admin')) return true

  // 段 2: OWNER_EMAIL 強制
  const ownerEmail = process.env.OWNER_EMAIL!.toLowerCase().trim()
  if (session.email?.toLowerCase().trim() === ownerEmail) return true

  // 段 3: ブレークグラス
  const breakGlass = ctx.req.header('x-admin-token')
  if (breakGlass && timingSafeEqual(breakGlass, process.env.ADMIN_WRITE_TOKEN!)) {
    log.warn('break-glass token used', { actor: session.email })
    return true
  }

  return false
}
```

- `timingSafeEqual` で timing attack を防ぐ
- ブレークグラス使用時は警告ログ + Slack 通知
- `OWNER_EMAIL` は Terraform で管理、変更は慎重に

### 1.4 セッション破棄（全サービス同時ログアウト）

`POST /api/auth/logout`:

1. Cookie からセッション無効化（Set-Cookie で max-age=0）
2. **Keycloak Backchannel Logout エンドポイント呼び出し** → 他サービス（VM, zen-track）にも伝搬
   - Keycloak Realm に各 OIDC RP の `backchannel_logout_url` を設定
   - Keycloak が各 RP の `/api/auth/backchannel-logout` へ POST（`logout_token` JWT 付き）
   - 各 RP は受信して自サービスのセッションも破棄
3. ブラウザ側で IndexedDB 完全削除（FE が `await indexedDB.databases().forEach(db => deleteDatabase(db.name))`）
4. メモリ上のセッションバウンド鍵を破棄

「このサービスだけログアウト」は提供しない（要件定義書 BR-19）。SSO Lv2 の一貫性を最優先。

各 OIDC RP は `/api/auth/backchannel-logout` を実装し、`logout_token` の検証後にローカルセッション破棄を行う必要がある。

### 1.5 IndexedDB 暗号化キャッシュ鍵の払い出し

`POST /api/cache-key`:

```typescript
app.post('/api/cache-key', authMiddleware, async (c) => {
  const user = c.get('user')

  // ランダム 256bit 鍵を生成
  const keyBuf = crypto.getRandomValues(new Uint8Array(32))
  const keyB64 = Buffer.from(keyBuf).toString('base64')

  // 監査ログ: 鍵払い出し
  await auditLog(c, {
    action: 'write',
    targetTable: 'session_cache_keys',
    details: { event: 'cache_key_issued' },
    visibleToUser: true,
  })

  // 鍵は DB には永続化しない（クライアントだけが持つ）
  return c.json({
    key: keyB64,
    validUntil: new Date(Date.now() + 24 * 60 * 60 * 1000).toISOString(),
  })
})
```

- 鍵は **クライアントだけが持つ**（ログアウトで完全消失）
- サーバが鍵を保持しないため、サーバ侵害時もキャッシュは復号不可能
- ただしユーザーがアクティブセッション中なら、トークンと一緒に鍵もメモリにある（passive read 防御のみ）

## 2. 暗号化実装の詳細

### 2.1 サーバ側暗号化（DB カラム）

対象: `text_entries.content`, `ai_messages.content`, `user_profiles.current_medications`, `user_profiles.allergies`, `oauth_tokens.encrypted_refresh_token` 等

#### 鍵管理

- マスター鍵を Secret Manager の `DATA_ENCRYPTION_KEY`（256bit）に保管
- 起動時に BE が取得、メモリで保持
- ローテーション: 半年に 1 回、新旧鍵の二重デコード期間あり（`key_version` 列を持たせる）

#### 暗号方式

- **AES-256-GCM**
- nonce は 12 バイトランダム生成、暗号文と共に格納
- 認証タグも暗号文に含める

#### Prisma 統合

`packages/db/encryption.ts`:

```typescript
export function encrypt(plaintext: string): Buffer {
  const nonce = crypto.randomBytes(12)
  const cipher = crypto.createCipheriv('aes-256-gcm', masterKey, nonce)
  const enc = Buffer.concat([cipher.update(plaintext, 'utf-8'), cipher.final()])
  const tag = cipher.getAuthTag()
  return Buffer.concat([nonce, tag, enc])  // 12 + 16 + N
}

export function decrypt(blob: Buffer): string {
  const nonce = blob.subarray(0, 12)
  const tag = blob.subarray(12, 28)
  const enc = blob.subarray(28)
  const decipher = crypto.createDecipheriv('aes-256-gcm', masterKey, nonce)
  decipher.setAuthTag(tag)
  return Buffer.concat([decipher.update(enc), decipher.final()]).toString('utf-8')
}
```

Prisma Middleware で `text_entries`, `ai_messages` 等の特定テーブルの特定カラムに対して自動 encrypt/decrypt。

### 2.2 クライアント側暗号化（IndexedDB）

`apps/web/lib/encryptedIndexedDB.ts`:

```typescript
export async function setEncrypted(key: string, value: unknown) {
  const sessionKey = await getSessionKey()  // メモリから
  const cryptoKey = await crypto.subtle.importKey(
    'raw', sessionKey, { name: 'AES-GCM' }, false, ['encrypt']
  )
  const nonce = crypto.getRandomValues(new Uint8Array(12))
  const enc = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv: nonce }, cryptoKey, encoder.encode(JSON.stringify(value))
  )
  const blob = new Uint8Array(nonce.length + enc.byteLength)
  blob.set(nonce); blob.set(new Uint8Array(enc), nonce.length)
  const db = await openDB()
  await db.put('cache', blob, key)
}
```

- 鍵はメモリ + IndexedDB メタ（`indexedDB-meta`）に保存。タブ間共有不要
- メタには `validUntil`, `userIdHash` も保存し、別ユーザーが同タブで使った場合の混線を防ぐ

## 3. 監査ログ実装の詳細

### 3.1 ログ書き込み

全 BE エンドポイントを覆う middleware:

```typescript
app.use('*', async (c, next) => {
  const startTs = Date.now()
  await next()
  const action = inferAction(c.req)  // read/write/delete/export
  const target = inferTarget(c.req, c.res)
  await db.auditLog.create({
    data: {
      eventAt: new Date(),
      actorType: c.get('user')?.isAdmin ? 'admin' : 'self',
      actorUserId: c.get('user')?.id,
      action,
      targetTable: target.table,
      targetId: target.id,
      targetUserId: target.userId,
      details: { method: c.req.method, path: c.req.path, durationMs: Date.now() - startTs },
      visibleToUser: target.userId === c.get('user')?.id || c.get('user')?.isAdmin,
    },
  })
})
```

### 3.2 ユーザー可視化

`GET /api/audit-logs/me`:

```sql
SELECT id, event_at, actor_type, action, target_table,
       LEFT(target_id::text, 6) AS target_id_short, details
FROM audit_logs
WHERE target_user_id = $1
  AND visible_to_user = TRUE
ORDER BY event_at DESC
LIMIT 50;
```

- 具体 `actor_user_id` は返さず、`actor_type` のみ（プライバシー）
- `target_id` は短縮（識別感は残しつつ詳細は隠す）

### 3.3 ログ自体のセキュリティ

- 監査ログ自体の改ざんを防ぐため、append-only（UPDATE/DELETE は管理者でも不可）
- Cloud SQL の権限分離: BE のサービスアカウントは INSERT のみ可、別の admin SA に DELETE 権を限定

## 4. AI コスト管理の詳細

### 4.1 コスト計算

各 AI Provider の単価表を `packages/shared/pricing.ts` に集約:

```typescript
export const PRICING = {
  'claude-opus-4-7': {
    inputUSDPer1M: 15,
    outputUSDPer1M: 75,
  },
  'gpt-4o': {
    inputUSDPer1M: 5,
    outputUSDPer1M: 15,
  },
  // ...
}
```

`POST /api/ai/chat` 完了時:

```typescript
const cost = (inputTokens * pricing.inputUSDPer1M + outputTokens * pricing.outputUSDPer1M)
  / 1_000_000 * USD_TO_JPY_RATE
await db.aiUsageLog.create({
  data: { userId, provider, model, inputTokens, outputTokens, costJpy: cost, purpose: 'chat' }
})
```

### 4.2 キャップ判定

`GET /api/ai/quota`:

```sql
SELECT COALESCE(SUM(cost_jpy), 0) AS used
FROM ai_usage_logs
WHERE user_id = $1
  AND date_trunc('month', created_at) = date_trunc('month', NOW());
```

`UserQuota` テーブル（または `user_preferences.feature_flags`）に `monthly_cap_jpy` を保持。デフォルト 1000 円、管理者が個別上書き可能。

### 4.3 キャップ enforcement

middleware:

```typescript
async function quotaMiddleware(c, next) {
  if (c.req.path.startsWith('/api/ai/')) {
    const { used, cap } = await getQuota(c.get('user').id)
    if (used >= cap) {
      return c.json({ error: { code: 'QUOTA_EXCEEDED', message: '今月の上限に達しました' } }, 422)
    }
  }
  await next()
}
```

縮退モデルへのフォールバック（Opus → Haiku）は MVP では実装せず、エラー返却のみ。

## 5. エラーハンドリング戦略

### 5.1 階層

```
Hono app.onError
  ↓ 既知のエラークラスに対応した HTTP ステータス
  ↓ クライアントには { error: {code, message, details}, requestId } 形式で返す
  ↓ Cloud Logging に詳細スタックを記録
```

### 5.2 エラークラス

`packages/shared/errors.ts`:

```typescript
export class ValidationError extends BaseError { code = 'VALIDATION_ERROR'; status = 400 }
export class UnauthorizedError extends BaseError { code = 'UNAUTHORIZED'; status = 401 }
export class ForbiddenError extends BaseError { code = 'FORBIDDEN'; status = 403 }
export class NotFoundError extends BaseError { code = 'NOT_FOUND'; status = 404 }
export class ConflictError extends BaseError { code = 'CONFLICT'; status = 409 }
export class QuotaExceededError extends BaseError { code = 'QUOTA_EXCEEDED'; status = 422 }
export class RateLimitedError extends BaseError { code = 'RATE_LIMITED'; status = 429 }
```

### 5.3 ユーザー向けメッセージ

- すべて日本語、専門用語を避ける
- 「AI」リテラルは使わない（旧仕様継続）
- 例: `QUOTA_EXCEEDED` → 「今月の利用量が上限に達しました。来月になると再びご利用いただけます。」

### 5.4 AI Provider のエラー

- 503 / 529（Anthropic 過負荷）: ユーザーには「混雑しています。しばらく後にお試しください」
- タイムアウト（30 秒超過）: 同上
- 自動リトライ・自動フォールバックは行わない（ADR-0006、複雑化禁止）

### 5.5 コストキャップ超過時

- `QUOTA_EXCEEDED` (422) を返す
- ユーザーメッセージ: 「今月のご利用上限に達しました。来月 1 日にリセットされます。」
- **縮退モデルへの自動切替は行わない**（CLAUDE.md フォールバックチェーン禁止）
- 管理者がユーザー個別キャップを上書きすれば再開可能

## 6. Keycloak Realm 設計

### 6.1 Realm 構成

- Realm 名: `cares-realm`（環境別: `cares-realm-dev` / `-staging` / `-prod`）
- すべての自社サービス（cares / VM / zen-track）が同じ Realm を共有

### 6.2 クライアント（OIDC RP）

| Client ID | 用途 | Access Type |
|---|---|---|
| `cares-web` | Next.js FE | confidential |
| `cares-api` | Hono BE（service-to-service） | bearer-only |
| `vm-web` | ValuationMatrix | confidential |
| `zen-track-web` | 将来 | confidential |

### 6.3 ロール

| Role | 説明 |
|---|---|
| `user` | デフォルト、全認証ユーザーに付与 |
| `admin` | 管理者権限 |
| `support` | 将来用、カスタマーサポート |

### 6.4 IdP フェデレーション

- **Google**: OIDC 設定（Google Cloud Console で client ID/secret 取得）
- 既存 ValuationMatrix の Google ログインユーザーは email でマッピング

### 6.5 認証フロー

- Browser flow: 通常ログイン（Google / Email+Password / Magic Link）
- Direct grant flow: 無効（OIDC のみ使う）
- Reset credentials flow: パスワードリセット
- First broker login flow: Google で初回ログイン時、cares 側プロファイル作成イベント

### 6.6 テーマ

- カスタムテーマで日本語化、cares ブランドカラー
- ログイン画面・パスワードリセット画面・同意画面をカスタム

### 6.7 Realm Export / Version 管理

- Keycloak Realm の設定を JSON エクスポート（`infra/keycloak/realm-cares-prod.json`）
- Git で版管理、変更は PR レビュー
- 適用は Keycloak Admin CLI + CI/CD

### 6.8 Backchannel logout 設定

- Realm 設定で各 OIDC RP に `backchannel_logout_url` を登録
  - `cares-web`: `https://cares.advisers.jp/api/auth/backchannel-logout`
  - `vm-web`: VM 側の同等エンドポイント
  - `zen-track-web`: 将来
- `backchannel_logout_session_required = true`
- 各 RP は受信した `logout_token` を JWKS で検証し、対応セッションを破棄

### 6.9 cares 単独ローンチ期間の挙動

VM の中央 IdP 移行が完了するまでは、cares のみ Keycloak に接続。この期間:

- ユーザーは「同じ Google アカウント」で cares と VM の両方にログイン可能
- ただし **セッション共有はなし**（VM は Identity Platform、cares は Keycloak）
- Backchannel logout も cares 単独で動作
- VM 移行後に真の SSO Lv2 が成立

## 7. テスト戦略

### 7.1 テストピラミッド

| 層 | 対象 | フレームワーク | 比率 |
|---|---|---|---|
| 単体 | Pure 関数、Zod スキーマ、暗号化、認可ロジック | Vitest | 70% |
| 統合 | BE エンドポイント、DB 含む | Vitest + Testcontainers (Postgres) | 25% |
| E2E | ブラウザ操作のシナリオ | Playwright | 5% |

### 7.2 単体テスト

- `packages/shared/*` は完全な型カバレッジ + 全ロジックのテスト
- 暗号化・復号は roundtrip テストを必須
- 認可 middleware の三段フェイルセーフ各パスをテスト

### 7.3 統合テスト

- 各 BE エンドポイントについて、Postgres 起動 + Prisma migrate + ハンドラ呼び出し
- 認可エラーケース、バリデーションエラーケースもカバー
- AI 呼び出しはモック（Anthropic SDK をスタブ）

### 7.4 E2E テスト

- 重要シナリオ: ログイン → 症状記録 → AI 解析 → エクスポート
- Playwright で実 Keycloak（テスト用 Realm）を使ってログインフローも検証
- staging 環境で smoke test を毎日実行

### 7.5 性能テスト

- k6 で API レイテンシを測定
- 想定負荷: ユーザー 50 人同時、5 req/min/user
- p95 < 500ms を目標

### 7.6 セキュリティテスト

- ZAP で OWASP Top 10 自動スキャン（CI で staging に対して）
- 認可ガード bypass テスト（他人の ID を URL に入れる等）
- ブラウザストレージへの PII 書き込み静的検査（ESLint カスタムルール）

## 8. デプロイ手順

### 8.1 CI/CD パイプライン

GitHub Actions:

```
main branch push
  ↓
1. Lint + Type check
  ↓
2. 単体テスト
  ↓
3. 統合テスト（Postgres Container 起動）
  ↓
4. Docker build (apps/web, apps/api)
  ↓
5. Artifact Registry に push
  ↓
6. Cloud Run staging にデプロイ
  ↓
7. E2E smoke test
  ↓
8. (手動承認)
  ↓
9. Cloud Run prod にデプロイ
  ↓
10. post-deploy smoke test
```

### 8.2 ロールバック

- Cloud Run の **トラフィック分割** で前バージョンに即時切戻し
- 履歴は 10 リビジョン保持
- DB マイグレーションが絡む場合は別途復旧手順（Prisma migrate reset 等）

### 8.3 Terraform 適用

- インフラ変更は `infra/terraform/` で PR
- `terraform plan` を PR 上で表示（OIDC で GCP 認証）
- マージで `terraform apply` 自動実行（環境別 workflow）
- 破壊的変更は手動承認必須

### 8.4 シークレット更新

- 通常運用: 管理画面 `/admin/keys` 経由で Secret Manager に書き込み（API 経由）
- 緊急時: GCP コンソールから直接更新可能
- Cloud Run は新バージョンデプロイ時に Secret Manager から最新値を取得

## 9. データ移行スクリプト設計

### 9.1 配置

`apps/migration/`：

```
apps/migration/
├─ src/
│  ├─ index.ts            # エントリ
│  ├─ users.ts            # Firebase Auth → Keycloak
│  ├─ firestore-to-pg.ts  # Firestore → Postgres
│  ├─ storage-copy.ts     # Firebase Storage → Cloud Storage
│  └─ verify.ts           # 差分確認
└─ Dockerfile
```

Cloud Run Job として実行。

### 9.2 Firebase Auth → Keycloak

```typescript
// 1. Firebase Admin SDK で全ユーザー取得
// 2. プロバイダ別に処理:
//    - Google: Keycloak Admin API で User 作成 + Federated Identity 追加
//    - Email+Password: User 作成 + 仮パスワード（=ランダム）+ 'reset password' email トリガ
//    - Anonymous: スキップ（個別対応）
// 3. user_link テーブルに global_uid を記録
```

### 9.3 Firestore → Postgres

各コレクションを順に処理:

```typescript
async function migrateSymptoms(uid: string, userId: string) {
  const snap = await firestore.collection(`users/${uid}/symptoms`).get()
  for (const doc of snap.docs) {
    const data = doc.data()
    await prisma.symptom.create({
      data: {
        id: data.id,  // 既存 ID 保持
        tenantId: userId,
        userId,
        recordedAt: data.createdAt.toDate(),
        conditionLevel: data.condition_level,
        sleepQuality: data.sleep_quality,
        memo: data.memo,
      },
    })
  }
}
```

- 暗号化対象カラム（text_entries, ai_messages）は移行時に暗号化
- 写真は Firebase Storage から Cloud Storage に `gsutil rsync`
- 全件処理後 `apps/migration/src/verify.ts` で count 一致を確認

### 9.4 並列度・性能

- ユーザー単位で並列化（10 並列程度）
- 大きなコレクションはバッチ化
- 失敗時はリトライ、最終的にユーザー別エラーログ

### 9.5 ロールバック

- 移行前に Firestore のスナップショットを Cloud Storage にダンプ
- 失敗時は DNS を旧 cares に戻す + Postgres は drop + 再試行

## 10. 開発環境セットアップ

### 10.1 必要なツール

- Node.js 22 LTS
- pnpm 9+
- Docker Desktop
- gcloud CLI
- Terraform
- Prisma CLI

### 10.2 ローカル起動

```bash
# 依存インストール
pnpm install

# .env 設定（.env.example を参照）
cp .env.example .env

# Docker Compose で Postgres + Keycloak 起動
docker compose up -d

# DB マイグレーション
pnpm db:migrate

# 開発サーバ起動（FE + BE 並行）
pnpm dev
```

### 10.3 Docker Compose（ローカル開発用）

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: cares
      POSTGRES_PASSWORD: cares
      POSTGRES_DB: cares_dev
    ports: [5432:5432]
    volumes: [pgdata:/var/lib/postgresql/data]

  keycloak:
    image: quay.io/keycloak/keycloak:25
    command: start-dev
    environment:
      KC_DB: postgres
      KC_DB_URL: jdbc:postgresql://keycloak-db:5432/keycloak
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: admin
    ports: [8080:8080]
    depends_on: [keycloak-db]

  keycloak-db:
    image: postgres:16
    environment:
      POSTGRES_DB: keycloak
      POSTGRES_USER: keycloak
      POSTGRES_PASSWORD: keycloak
    volumes: [kcdata:/var/lib/postgresql/data]

volumes:
  pgdata:
  kcdata:
```

### 10.4 推奨 IDE 設定

- VS Code + Prisma 拡張 + ESLint + Prettier
- TypeScript strict mode
- 共通 `.editorconfig`

### 10.5 開発者向け規約

- コミット前に `pnpm lint && pnpm typecheck && pnpm test` 必須（pre-commit hook）
- localStorage / sessionStorage への書き込みは ESLint で禁止
- Prisma スキーマ変更は別 PR、レビュー必須
- 暗号化対象カラムを増やす際は `packages/db/encryption.ts` の対象リストを更新
