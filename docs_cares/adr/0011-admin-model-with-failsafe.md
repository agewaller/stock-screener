# ADR-0011: 管理者モデル（三段フェイルセーフ）

- Status: Accepted
- Date: 2026-05-26
- Deciders: オーナー、Claude（提案）

## Context

旧仕様の絶対原則として「管理者 `agewaller@gmail.com` を絶対にロックアウトしない」がある。設定ミスや認証基盤の障害で管理者が自分のサービスから締め出される事故は、これまで何度か対策コードを足してきた経緯がある。

新仕様では Keycloak を中央 IdP（ADR-0002）として導入するため、管理者識別を Keycloak ロールベースにすると Keycloak 設定ミスで再びロックアウトのリスクが生じる。

複数管理者の運用も視野に入れる（オーナー以外への委譲）。

## Decision

### 三段フェイルセーフ

「オーナーを絶対にロックアウトしない」原則を 3 つの独立機構で担保する:

#### 1. 通常運用: Keycloak ロール `admin` で識別

- OIDC トークンの role claim を BE が検証
- 複数管理者をサポート（オーナー以外にも admin 付与可）
- 「フロントエンド・バックエンドの admin 判定」はこの方式が標準

#### 2. 追加保険: オーナーメール強制管理者

- BE 側に `OWNER_EMAIL` 環境変数（デフォルト `agewaller@gmail.com`）
- OIDC `email` クレームが OWNER_EMAIL に一致したら、Keycloak ロールに関係なく強制的に admin 扱い
- Keycloak 設定ミスでロールが消えてもオーナーは管理画面に入れる

#### 3. ブレークグラストークン

- 旧仕様の `ADMIN_WRITE_TOKEN` 相当を継続
- Secret Manager に保管
- Keycloak 完全障害時の最終手段
- HTTP ヘッダ `x-admin-token` で BE を一時的に開けられる
- 通常ログインフローでは使わない（緊急時のみ）

### 複数管理者の制約

- Keycloak ロール `admin` は任意のユーザーに付与可能
- ただし `OWNER_EMAIL` のロール変更は **管理 UI からは不可**（誤って自分から admin を剥がせない）

### 管理者向け機能（旧仕様継続）

- プロンプト管理（CONFIG の AI プロンプトを動的編集）
- API キー管理（Secret Manager 経由、画面では「設定済み true/false」のみ表示。**鍵値は決して返さない**）
- データ管理（ユーザー一覧・個別データ閲覧・エクスポート）
- 監査ログ閲覧

### 管理操作の監査ログ

- すべてユーザー本人にも公開（ADR-0007 の監査ログ原則と整合）
- 「管理者があなたのデータに read/write/export した」が見える

### 認可ガード

すべての管理 API は共通 middleware で「Keycloak ロール OR オーナーメール OR ブレークグラストークン」の三重チェックを実装。

### 管理画面の 2FA

- ADR-0002 で 2FA はユーザー任意としたが、`admin` ロール持ちには 2FA 強制を検討
- 強制とした場合、Keycloak の Conditional Authenticator 機能を使う

## Consequences

### 良い面

- 単一障害点（Keycloak 設定ミス、Keycloak 障害）でオーナーがロックアウトされない
- 複数管理者で運用可能（業務委譲が可能）
- ブレークグラスは緊急時のみ、通常運用に影響しない
- 旧仕様の運用ノウハウ・トラウマを反映した堅牢な設計

### 悪い面

- 認可ガードに 3 つの分岐があり、コードが少し複雑
- ブレークグラストークンの管理が必要（Secret Manager のみで、誤って漏れないように）
- `OWNER_EMAIL` 環境変数の変更は Terraform で慎重に管理する必要がある

### 派生タスク

- 認可 middleware の実装（フェーズ 4）
- ブレークグラス使用時の警告ログ・Slack 通知
- `OWNER_EMAIL` 環境変数の Terraform 管理ルール
- `admin` ロール持ちの 2FA 強制方針確定

## Alternatives Considered

- **Keycloak ロールのみ**: 設定ミスで自分から admin を剥がしたらロックアウト。過去の事故と同じパターン。
- **オーナーメールのみハードコード**: 旧仕様と同様だが、複数管理者を委譲できない。
- **ブレークグラスなし**: Keycloak 完全障害時に管理操作が一切不可能。
- **常に三重チェックではなく、Keycloak 障害時の自動切替**: 自動切替ロジック自体がバグの温床。常時三重チェックの方が信頼性が高い。
