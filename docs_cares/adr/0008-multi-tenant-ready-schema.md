# ADR-0008: マルチテナント対応スキーマ（MVP は B2C）

- Status: Accepted
- Date: 2026-05-26
- Deciders: オーナー、Claude（提案）

## Context

cares は現在純粋な B2C（個人向け）サービス。将来的に医療機関・介護事業者・企業契約等の B2B 展開、および家族・医師・介護チームとの限定共有を実装する可能性がある。

スキーマ設計時点で `tenant_id` 等のテナント識別子を入れておくか、後付けで全テーブル ALTER するかは大きな分岐点。後者は数十テーブルへの一括変更となり、データ移行・アプリ修正のコストが大きい。

## Decision

### 軸 A: 法人展開（B2B）

**将来 B2B を見据えた設計、MVP は B2C**。

- すべてのユーザーデータテーブルに **`tenant_id` カラムを含める**
- MVP では実質的に `tenant_id = users.id`（個人テナント）の 1:1 紐づけ
- 将来 B2B 化する際は、tenant_id に医療機関等の組織を割り当て、その配下にユーザーをぶら下げる構造に拡張

### 軸 B: データ共有・コラボレーション

**家族・医師との限定共有を MVP 後に実装**。

- MVP: 本人と管理者のみ閲覧
- 次フェーズ: ユーザーが指定した家族 / 医師 / 介護関係者を招待 → 範囲限定・期間限定で閲覧権限付与
- 共有の細分粒度: カテゴリ別（症状のみ等） / 期間別 / 読み取りのみ or コメント可

### 権限モデル設計（MVP 後実装、設計はフェーズ 3）

- ロール: `owner` / `viewer` / `commenter` / `admin`
- スコープ: カテゴリ別 / 期間別 / レコード別
- 招待フロー: メール招待 + Magic Link で簡易ログイン
- 監査ログに「誰の権限で誰のデータを見たか」を残す

### B2B 拡張時の追加要件（将来）

- 法人管理画面（テナント管理者がユーザー一覧・利用状況を見られる）
- 法人課金（個人課金とは別）
- SCIM 等の企業ユーザー一括プロビジョニング

これらは MVP では実装しない。

## Consequences

### 良い面

- 将来 B2B 化する際にスキーマ変更が最小化
- 共有機能を加えても tenant 設計がベースになる
- アクセス制御を `tenant_id` ベースで一貫実装可能（RLS / アプリ層ガード）
- ADR-0002 の SSO とも整合（global_uid と tenant_id を組み合わせた認可）

### 悪い面

- MVP では使わないカラムが全テーブルに存在（軽い無駄）
- アプリ層で「現在の tenant_id」をスコープする実装が必要（B2C 時は単純だが間違えやすい）
- RLS（Postgres Row Level Security）を使うかアプリ層で実装するかの選択が必要

### 派生タスク

- スキーマ設計時の `tenant_id` 配置ルール明文化（フェーズ 3）
- アクセス制御の実装方針（RLS vs アプリ層ガード）の決定
- 招待フロー UX 設計（MVP 後）
- B2B 展開時の課金設計（将来）

## Alternatives Considered

- **純粋 B2C（tenant_id 不要）**: シンプルだが、将来 B2B 化時に数十テーブルへの破壊的変更が必要。
- **最初から B2B 対応**: 機能スコープが膨らみ、MVP リリースが遅延。需要が確定していない段階で実装するのは早すぎる最適化。
- **将来 schema 分けて運用**: tenant が増えるたびにスキーマを増やす方式。運用負荷大、ジョインクエリ困難。
