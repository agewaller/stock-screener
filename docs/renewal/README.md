## なぜリニューアルするのか（一行サマリ）

**マルチデバイスからログインしても同じアカウントとして使えるようにするため、
ブラウザ内ストレージを「データの真値」として扱う設計をやめ、サーバサイド DB（Firestore）を一元的な真値に置く。**

現状は localStorage が真値・Firestore がレプリカという構造で、結果として:

- 同一ユーザーが端末 A と端末 B で別物の状態を持つ
- ログアウトしても致命的トークン群（API キー / Fitbit / Google Calendar）が残置される
- 医療 PII（要配慮個人情報）が無暗号で端末側に常時存在する
- API キーが全ユーザー共有・平文・DevTools 読取可能

詳細は [adr/0001-cloud-first-data-management.md](adr/0001-cloud-first-data-management.md) を参照。

## リニューアルのスコープ（決定済）

決定済の方針:

- **主軸**: ブラウザローカル管理 → サーバサイド DB 管理への刷新（マルチデバイス対応）
- **提供形態**: 現状維持（SPA on GitHub Pages、vanilla JS）
- **進め方**: まず現状凍結スナップショットを作成 → 新仕様要件 → 移行計画 → 実装

未決定（次のヒアリング待ち）:

- オフライン対応の方針（完全停止 / read-only / 限定書込キュー）
- ゲストモード（未ログイン入力）の扱い
- API キー保存先のリストラクチャ（管理者専用に閉じる / per-user Firestore のみ / Worker 注入のみ等）
- 写真の保存先（Firestore は不可 / Cloud Storage / 端末のみ）
- 既存ユーザーの localStorage データ移行手順
- v1.0 → v2.0 切替時のダウンタイム / フィーチャーフラグ
- API 課金管理（apiUsage 5000 件の継続要否）
