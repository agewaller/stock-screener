最終更新: 2026-05-06

ソースコード（`js/*.js` 13モジュール / `sw.js` / `index.html`）を一次資料として
直接走査し、ユーザーのブラウザ内に保存される全データを列挙したリファレンス。

姉妹文書: [データ構造.md](データ構造.md)（Store / localStorage / IndexedDB / Firestore / KV の全体図）。

## 0. 全体俯瞰

本アプリは GitHub Pages 上の vanilla JS SPA であり、サーバ側セッションを持たない。
そのためユーザー固有データ・認証トークン・グローバル設定の **すべてがブラウザ内ストレージに集中している**。

| 層 | 用途 | 永続性 | 容量目安 | ソース |
|---|---|---|---|---|
| **localStorage** | 状態スナップショット + 認証/設定/トークン | ブラウザ次第（基本永続） | ~5–10MB | `js/store.js`, 各モジュール直接 |
| **IndexedDB** | 写真 dataURL / オフライン書き込みキュー | 永続 | ~50MB+ | `js/idb.js` |
| **Cache Storage** | 静的アセット（SW） | デプロイで更新 | ~10MB | `sw.js` |
| **sessionStorage** | （未使用） | — | — | — |
| **Cookie** | （未使用、Firebase Auth は IndexedDB 永続化） | — | — | — |

## 1. localStorage（A）— Store クラス経由の `cc_*` キー

`js/store.js:132-155` の `PERSIST_KEYS` に列挙された **44 フィールド**（+ `cc_schema_version`）が

## 2. localStorage（B）— Store 外の直接アクセス

`grep localStorage\.` で **91 箇所** ヒット。Store 経由を除いた直接アクセスを以下に整理。
`cc_*` 接頭辞で書かれたキーも一部はこちら（Store 外管理）。

### 2-A. システム設定（device-shared）

| キー | 書込 | 意味 |
|---|---|---|
| `firebase_config` | `firebase-backend.js:942` | Firebase 初期化用 JSON。Admin が管理 UI から投入し、以降全ユーザーが共有 |
| `anthropic_proxy_url` | `firebase-backend.js:450` / `:664` / `:755` | Cloudflare Worker プロキシ URL（既定 `stock-screener.agewaller.workers.dev`） |

| `enable_shared_guest_ai` | （UI 経由） | ゲスト共有 AI の ON/OFF（`ai-engine.js:483` で参照） |
| `admin_emails` | `app.js:4875,4890` | 追加管理者メール JSON 配列（ハードコードの `agewaller@gmail.com` は除外） |

### 2-B. API キー（admin-managed, **全ユーザー共有**, 平文）⚠️

```
firebase-backend.js:445-447 / 658-660 / 740-742
```

| キー | 用途 |
|---|---|
| `apikey_anthropic` | Claude API（プロキシ未使用時に直接叩く） |
| `apikey_openai` | OpenAI / GPT-4o |
| `apikey_google` | Gemini / Google Vision |

設計意図: **Firestore `/admin/config` に管理者が投入したキーを起動時に localStorage へ降格** し、
クライアントから直接 LLM API を叩くか or Worker プロキシ経由で叩く。

> **構造的な含意**
> - DevTools で誰でも生キーが見える。
> - `clearAll()` の `PRESERVE_KEYS`（`store.js:381-383`）に含まれているため、ログアウト後も端末に残る。
> - 共用デバイスでは「前のユーザーが置いていったキー」を後続ユーザーが利用可能。

## 3. IndexedDB — `health-diary` DB v1

定義: `js/idb.js:6-7`。サーバー側に置かない大型バイナリと、
Firestore オフライン時の書き込みバッファを保持。

| Object Store | キー | 用途 |
|---|---|---|
| `images` | `id` | 写真 base64 dataURL（`{id, dataUrl, savedAt}`）。localStorage 容量回避 |
| `largeData` | `key` | 予約済み・**現在未使用**（レガシー枠、削除候補） |
| `pendingWrites` | autoIncrement / `collection` index | オフライン時の Firestore 書き込みキュー（復帰時 flush） |

> **構造的観察**: Firestore SDK 自体も `enablePersistence()`（`firebase-backend.js:31`）で
> IndexedDB に独自キャッシュを取る。アプリ独自の `pendingWrites` と SDK のキャッシュは **別物**。

## 4. Cache Storage — Service Worker（`sw.js`）

`health-diary-v1-{COMMIT_HASH}` という単一キャッシュに
`index.html` + `js/*.js` 13モジュール + `manifest.json` をプリキャッシュ。
COMMIT_HASH は GitHub Actions で置換されるため、デプロイごとに新キャッシュへ切り替わる。
外部 API（`*.googleapis.com` 等）は対象外（`sw.js:44`）。

## 5. データの「重複」と「降格」の構造

健康日記は **同じデータが複数層に複製される設計** が多い：

1. **API キー：Firestore `/admin/config` → localStorage（平文）**
   起動時降格。Firestore Rules で読取制御するはずがクライアントに落とすため、
   端末側に常時平文で存在する。

2. **健康記録：Store(localStorage) ↔ Firestore（ログイン時のみ）**
   Store が真値、Firestore がレプリカ。オフライン時は `pendingWrites` を経由。

3. **写真：localStorage に id 配列のみ + IndexedDB に実体**
   `cc_photos` が IndexedDB レコードを参照。クリーンアップ漏れに注意。

4. **分析履歴：`cc_latestAnalysis` ⊆ `cc_analysisHistory[0]`**
   先頭要素は重複している。

5. **カレンダー：`cc_calendarEvents` + `ics_calendar_url` + `google_calendar_*` + Firestore `/users/{uid}/private/calendar`**
   経路が多い。disconnect 時のクリーンアップ範囲が不明瞭。

## 6. PII / 機微情報の所在

| 種別 | 保存層 | キー / フィールド | 暗号化 |
|---|---|---|---|
| 認証情報（メール・氏名・写真URL） | localStorage | `cc_user.{email,displayName,photoURL}` | なし |
| 医療記録（症状・服薬・血液検査） | localStorage | `cc_symptoms` / `cc_medications` / `cc_bloodTests` 他 | なし |
| 自由記述・音声書き起こし | localStorage | `cc_textEntries`, `cc_plaudAnalyses` | なし |
| プロフィール（年齢・身長・体重） | localStorage | `cc_userProfile` | なし |
| カレンダータイトル・本文 | localStorage | `cc_calendarEvents` | なし |
| 写真の実バイト | IndexedDB | `images` ストア | なし |
| OAuth トークン | localStorage | `google_calendar_access_token`, `fitbit_token` | なし |
| LLM API キー | localStorage | `apikey_*` | なし |

`Privacy.anonymizeText()`（`privacy.js:30-`）は AI 送信**直前**のみ作用するため、
ストレージ自体の PII は一切マスクされない。かつ既定 OFF。

## 7. ログアウト時の残置（`store.clearAll()` の挙動）

`store.js:373-394` の `PRESERVE_KEYS` に基づき、ログアウト後も以下が残る：

```
firebase_config / anthropic_proxy_url / anthropic_mode / admin_emails /
enable_shared_guest_ai / apikey_anthropic / apikey_openai / apikey_google /
ics_calendar_url / google_calendar_oauth_connected /
fitbit_token / fitbit_client_id / fitbit_refresh_token /
apple_health_last_import / plaud_email
```

加えて `localStorage.clear()` で消えても **IndexedDB は別途保持**
（`images` / `pendingWrites` も残る点に注意）。
コメント上の意図は「再ログイン時の再設定を不要にする」だが、
共用端末・端末譲渡時のシナリオは未考慮。

## 8. 構造的観察（改修議論の種）

1. **API キーの平文 localStorage 永続化**（`store.js:381-383`, `firebase-backend.js:445-447`）
   → DevTools 露出 + ログアウト後残置 + 共用端末で前ユーザーのキーが再利用される。
2. **PII 匿名化が opt-in & 既定 OFF**（`privacy.js:9-10`）
   → 匿名化対象もストレージ自体ではなく送信直前のみ。ストレージ側の暗号化は無し。
3. **`largeData` オブジェクトストアが未使用**（`idb.js`）
   → レガシー枠。削除可能か確認要。
4. **`cc_latestAnalysis` と `cc_analysisHistory[0]` の重複**
   → 整合性ロジックが Store にあるか、ある時点で乖離するか要検証。
5. **「device-shared API キー」と「per-user OAuth トークン」が同一 PRESERVE_KEYS に同居**
   → 設計上の役割分離が崩れている。
6. **IndexedDB `pendingWrites` の retry / 失敗時挙動**
   → 詳細追加調査が必要。
7. **`cc_*` 命名の混在**（Store 経由は `cc_{PERSIST_KEYS の名}`、別系統は `cc_language` `cc_errorLog` `cc_affiliate_tracking`）
   → 一目で「Store 管理」か否か判定できない。

## 9. 検証方法

ブラウザの DevTools で以下を開けば本書の各キーが確認できる：

- `Application` → `Storage` → `Local Storage` → `https://cares.advisers.jp`
- `Application` → `Storage` → `IndexedDB` → `health-diary` → `images` / `largeData` / `pendingWrites`
- `Application` → `Storage` → `Cache Storage` → `health-diary-v1-*`

ローカル開発時:

```
npm start
# → http://localhost:8080 で同様に確認
```

機械的な再列挙には:

```
grep -rn "localStorage\." js/ index.html
grep -A2 "PERSIST_KEYS" js/store.js
grep -A2 "PRESERVE_KEYS" js/store.js
```
