# 健康日記（Cares）― 製品詳細

- **一言**: 慢性疾患患者に**寄り添いながら、蓄積データから「最適・最新・something new」な気づきと次の一手を返す**体調記録・情報整理ツール。
- **本番サイト**: https://cares.advisers.jp
- **リポジトリ**: `agewaller/stock-screener`（現行稼働＝vanilla JS SPA / GitHub Pages）、`agewaller/cares`（GCP リアーキ版＝Next.js+Hono+PostgreSQL、機能移植中）
- **対象**: toC（慢性疾患の当事者と家族）／ **収益**: toC 課金（無料お試し→月額）

## 1. ビジョン・顧客価値

慢性疾患（膠原病・糖尿病・自己免疫疾患・精神疾患など多数）の患者は、**症状・薬・受診の経過をうまく説明できない**という共通の困りごとを持つ。健康日記は記録を溜め、AI が「伝わる言葉」に変換する。

**健康日記固有の価値＝「寄り添い × 最適・最新・something new」。** 単なる記録の反復でも一般論でもなく、その人の蓄積データを根拠に「いまのあなたに最適で、最新で、ひと味違う気づき」と**今日試せる具体的な次の1アクション**をそっと返す（実装: 日次解析/投稿即時コメントのプロンプトが「寄り添い＋最適化＋SOMETHING NEW」を必須化）。同じ助言の繰り返しではなく、変化に合わせて提案そのものが進化する。

**提供価値:**
- その人の蓄積データから、**最適・最新・something new な気づき＋次の1アクション**を返す（認知を変えて行動を促す）
- 通院時に「前回からの経過」を医師へ的確に伝えられる（医師向け要約）／家族向けのやさしい要約も自動作成
- バラバラな体調・薬・食事・睡眠の記録を一元化し振り返れる
- **データは患者本人のもの**（編集・削除・書き出し自由）という安心

> 補足: cares にも「意識の焦点（7レイヤー意識）」があるが、これは**ユーザーが自分でその日の意識の層を記録するタグ項目**（＋平易ラベル表示）であり、発話から意識状態を**解析**するのは zentrack 側の役割（[03-zentrack.md](03-zentrack.md)）。

**ペルソナ**: 65歳女性（専門用語を避けた UI）。実際の情報収集は娘世代（40-50代）が担うことも多い。

## 2. 詳細な機能

### 記録（ジャーナリング）
- **記録カテゴリ（約8〜10種）**: 体調・症状・薬・食事・睡眠・数値（体温/血圧など）・写真・テキストメモ 等
- **編集・削除・復元**: 1件単位で編集削除でき、ソフト削除→30日後ハード削除→復元APIあり（即時ハード削除ではない）
- **ドラフト autosave**（drafts テーブル）
- **疾患選択**: 疾患に応じた記録項目・プロンプトの出し分け

### AI 分析
- **AI 日次分析**: その日の記録への振り返り
- **AI 深堀り**: 蓄積データからの掘り下げ
- **AI レポート**: 受け手別（医師/家族/共有用）の要約生成
- **AI 研究（リサーチ）**: PubMed 等の根拠探索
- **お試し AI 分析（FN-TRIAL-01）**: 登録前に体験できる入口（CVの起点）
- **プロンプト DB-driven**: AI 応答はすべて DB のプロンプトで制御（コードにハードコードしない）
- **月次キャップ**: AI コストの月額上限（超過は 422 で拒否）

### 管理・運用
- **管理画面（FN-ADMIN-01〜07）**: プロンプト管理・APIキー・データ管理・キャンペーン
- **監査ログ**
- **エクスポート**: JSON（CSV/PDF/FHIR は新版で計画中）

### 統合（計画/一部）
- Plaud（音声）/ Google Calendar / Fitbit / Apple Health / 専門家メール

## 3. 技術・アーキテクチャ

| | 現行（stock-screener） | 新版（cares） |
|---|---|---|
| 形態 | vanilla JS SPA（ビルド無し、`js/*.js` 13モジュール） | モノレポ（pnpm）：Next.js 15 + Hono + Prisma |
| ホスティング | GitHub Pages（cares.advisers.jp） | GCP Cloud Run + Cloud SQL(PostgreSQL) + GCS + Secret Manager |
| 認証 | Firebase Auth | GCP Identity Platform（Firebase Auth）Google + Email/Password |
| AI | Cloudflare Worker 経由（鍵はサーバのみ） | サーバ側のみ（BYOK不可・フォールバック連鎖禁止） |

**設計の核**: データ主権5原則／ブラウザに PII を置かない（localStorage禁止）／DBフィールド暗号化(AES-GCM)／AI鍵はサーバ側のみ／管理者ロックアウト防止（三段フェイルセーフ、OWNERはemail/passwordログイン時のみadmin）。

## 4. マーケティング戦略（低予算・オーガニック中心）

**ポジショニング**: 「記録するほど、あなたの体調が"伝わる言葉"になる。データはあなたのもの。」

- **疾患別 SEO コンテンツ**: 「○○病 症状 記録/日記」等。stock-screener には既に多数の疾患別 HTML ページ（diabetes/asthma/depression…）があり SEO 資産になる
- **当事者インタビュー**（note / YouTube）
- **患者会・疾患コミュニティ**への地道なリーチ（広告より信頼が効く領域）
- **お試しAI分析を入口**にウェイトリスト/メルマガで関係構築
- 広告は P3 まで最小（疾患名キーワードの少額検索広告で CV テストのみ）

**規制の注意**: 医療広告ガイドライン/薬機法。「診断・治療」を謳わず「記録・整理ツール」として訴求する。

## 5. 収益モデル（toC）

- 無料お試し（お試しAI分析・基本記録）→ 継続者向け月額プラン（価格は検証中）
- 将来：家族共有プラン、医療機関/患者会との連携プラン等を検討余地
