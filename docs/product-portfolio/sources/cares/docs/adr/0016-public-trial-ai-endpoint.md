# ADR-0016: ログイン不要のお試し AI 分析（公開 AI エンドポイント）

- Status: Accepted
- Date: 2026-06-02
- Deciders: オーナー、Claude（提案）

## Context（背景）

[Issue #27](https://github.com/agewaller/cares/issues/27): 現在はログインしないと日記も書けず AI 分析も
受けられない。**ログイン前のトップ画面で日記を書いて「お試し分析」できる機能**が求められた。

- 書き込まれた日記データは保存不要（紐づけるユーザがいない）
- 適用するプロンプトは、日記文章を分類してその結果から選択する
- 処理時間は最大 15 秒以内

これは cares で初めての**認証なしで AI を呼び出せる公開エンドポイント**となる。
AI キーはサーバ側にあるため（ADR-0006）キー漏洩のリスクはないが、**第三者が無制限に
AI コストを発生させられる**という新しい悪用リスクが生まれる。本 ADR はその対策を含めた
設計を記録する。

## Decision（決定）

### エンドポイント

**`POST /api/trial/analyze`** — 認証ミドルウェアを skip する唯一の AI エンドポイント
（既存の skip は `/api/healthz` のみ）。

- リクエスト: `{ "text": "<日記文章>" }`（最大 4000 文字）
- レスポンス: `{ category, comment, modelUsed, remainingToday }` + `Cache-Control: no-store`
- **日記本文は DB に一切保存しない**（コスト記録のみ。ADR-0007 のブラウザ PII 禁止と同じ思想で、
  サーバ側にも紐づけ先のない PII を残さない）

### AI 2 段呼び出し（分類 → 分析）

1. **分類**: Haiku（`max_tokens: 16`、タイムアウト 4 秒）で日記文を 9 カテゴリ
   （記録 8 カテゴリ + other）の単語 1 語に分類
2. **プロンプト選択**: 分類結果 → DB の `trial_{category}` プロンプト → 無ければ `trial_general`
   （プロンプト解決のカスケードであり、プロバイダのフォールバック連鎖ではない = ADR-0006 と整合）
3. **分析**: Haiku（`max_tokens: 1024`、タイムアウト 9 秒）で分析コメント生成。
   ADR-0013 S-7 に従い**必ず「次の 1 アクション」を 1 つ含める**

時間予算: 分類 4 秒 + 分析 9 秒 + オーバヘッド 2 秒 = 15 秒以内。超過時は 504 を返す。

### 悪用・コスト対策（三層）

| 層 | 内容 | 超過時 |
|---|---|---|
| 入力制限 | テキスト最大 4000 文字（トークン量と処理時間を抑制） | 400 |
| IP レート制限 | **1 IP あたり 1 日 3 回**。生 IP は保存せず SHA-256(IP + salt) で記録 | 429 |
| 全体月次キャップ | **お試し全体で月 3000 円**（オーナー個人の 1000 円/月キャップ [BR-20] とは完全別枠） | 422 |

- IP ハッシュ用 salt は `secrets` テーブルに `TRIAL_IP_SALT` として保存（初回に自動生成）
- コストは新テーブル `trial_usage` に記録（`ai_usage_logs` は userId 必須のため使わない =
  認証済みユーザの課金管理に一切影響しない）
- レート制限は新テーブル `trial_rate_limits`（ipHash + 日付の複合キー、古い行は随時削除）

### プロンプトは DB 駆動

`trial_classify` / `trial_general` / `trial_{category}` ×8 を `extra-prompts.json` から seed。
管理画面（FN-ADMIN-01）で編集可能。

### モデル

分類・分析とも **Haiku 固定**（`DEFAULT_MODEL_ID`）。お試しユーザにモデル選択は提供しない
（コスト管理と 15 秒制約のため）。

## Consequences（含意）

### 良い面

- 未ログインの見込みユーザが製品価値（AI 分析）を体験でき、登録への導線になる
- 悪用されてもコスト上限は月 3000 円で頭打ち（オーナー個人の AI 枠には影響しない）
- 日記本文を保存しないため、個人情報保護の観点で最もクリーン
- 認証済みフロー（quick-entry / analysis 等）への変更はゼロ（ミドルウェアの skip 1 行のみ）

### 悪い面・トレードオフ

- IP 制限は VPN / モバイル回線の IP 変更で回避可能（全体キャップが最終防衛線）
- 共有 IP（社内 NAT 等）では複数人で 1 日 3 回を分け合うことになる
- Cloud Run のインスタンス間でレート制限を共有するため DB を使う = お試し 1 回につき
  DB 書き込みが発生する（負荷は軽微）
- お試し枠が悪用で使い切られると、正規の見込みユーザが月末まで試せない

### 後で再検討すべきこと

- reCAPTCHA / Turnstile の導入（ボット対策が必要になったら）
- お試し回数・コストのダッシュボード（管理画面への統計表示）
- 多言語対応（現状は日本語のみ）

## Alternatives Considered（検討した代替案）

- **Firebase 匿名認証（`signInAnonymously`、FN-AUTH-03）を実装して使う**: 匿名ユーザにも
  uid が発行され既存のコスト管理に乗るが、Issue の要件（保存不要・お試し）に対して過剰。
  匿名→正規昇格（FN-AUTH-08）等の未解決論点（ADR-0013 S-8）を先に解く必要があり重い。
- **キーワード分類（既存 `inferAutoCategory()`）**: 瞬時・無料だが、オーナーの意向により
  キーワードにない表現も拾える AI 分類を採用（処理時間は 15 秒以内に収まる）。
- **`ai_usage_logs` に system ユーザ行を作って記録**: 既存コードを最大限再利用できるが、
  users テーブルに偽ユーザが混ざり、ユーザ統計・監査の意味が濁る。専用テーブルで分離。
- **レート制限なし（全体キャップのみ）**: 実装は最小だが、1 人が短時間で枠を使い切ると
  他の見込みユーザが試せなくなるため不採用。
