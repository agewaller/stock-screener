# ホーム記帳のワンページ化 — 再設計プラン (issue #33)

> ステータス: **計画 (proposal)**。実装着手前のたたき台。確定したら SC-02 / SC-03 に反映する。
> 出典: staging 実機テスター 2 名（Haruka さん / Frances H さん）フィードバック（LINE, 2026-06）

## 1. 背景 (なぜ)

実機テスターから、ホームの入力導線が分かれていて迷う・心理的ハードルが高い、という声が複数。

- 「2 つのセクションをあえて分けた理由は？ ワンページ構成に設計し直せないか。その方が操作の心理的ハードルが下がる」(Frances H ②)
- 「食事・睡眠・体重・心拍数・血圧などの基礎データは 1 ページに統合し、タップだけで入力が完結する仕様が理想」(Frances H ④)
- 関連: #1（トップ即時記帳）, #33（本 issue）, #35（義務的記録からの脱却）

## 2. 現状の問題 (SC-02 ホーム)

現在のホーム（`apps/web/components/QuickPost.tsx` ＋ ダッシュボード）には**入力の入口が 2 つ並存**している:

```
┌─ ① いま、どうですか？ ───────────┐   ← 即時記帳 (QuickPost / /api/quick-entry)
│  [自由記述テキストエリア]          │
│  📷 写真を添える   [ 記帳する ]    │
└──────────────────────────────┘
┌─ ② ＋ 今日の体調を記録する ───────┐   ← 構造化記録 (SC-03 へ遷移)
│  📷 写真を追加     📝 メモを追加    │
└──────────────────────────────┘
```

- **どちらに書けばいいか分からない**（①と②の役割の差が利用者に伝わらない）。
- ②は別画面（SC-03）への遷移で、入力完了までの手数が多い。
- 基礎データ（バイタル等）が数値の手入力前提で、brain fog / 筋萎縮のある利用者に重い（#34 とも関連）。

## 3. 設計方針

1. **入口は 1 つ**。「まず書く／撮る」を最上段の単一フォームに集約し、即レス（#1 の体験）を維持。
2. **構造化入力は"後出し・任意"**。自由記述から自動仕分け（既存 `inferAutoCategory`）した結果を、その場で「タグの確認・微調整」としてインライン表示。別画面遷移を必須にしない。
3. **基礎データはタップ完結**。よく使う値をチップ/ステッパーで選べるようにし、キーボード入力を極力不要に。
4. **1 スクロールで完結**。ホームを「書く → ひとこと返る → 必要なら数値を足す」の縦 1 本に。

## 4. 提案レイアウト (ワンページ)

```
┌─ いま、どうですか？ ───────────────────────┐
│ [なんでも書ける自由記述 ............................] │
│ 📷 写真   🎙 音声(→#34)            [ 記帳する ]      │
├─ (送信後インライン展開) ──────────────────────┤
│ 💬 いまのひとこと: 「…寄り添いコメント…」            │
│ 🏷 仕分け: [気分▾]  ← タップで修正可                  │
│ ＋ 数値も残す:  😴睡眠 [選択] ⚖️体重 [選択]           │
│                🩺血圧 [選択] ❤️心拍 [選択]            │
│   (タップで開く・任意・スキップ可)                    │
└────────────────────────────────────────────┘
```

- 上段は常時 1 つだけ。②「今日の体調を記録する」ボタンは廃止し、機能はインラインの「＋数値も残す」に吸収。
- 「＋数値も残す」は**チップ/ステッパー中心**（例: 睡眠は「よく眠れた/普通/眠れない」＋時間プリセット）。自由入力も可能だが既定はタップ。
- 過去日付（#30）・一括取り込み（#31）への導線は折りたたみメニューに置き、通常フローを汚さない。

## 5. 実装ステップ (段階移行・後方互換)

| 段階 | 内容 | 影響範囲 |
|---|---|---|
| S1 | ②「今日の体調を記録する」ブロックを撤去し、QuickPost を単一入口に。SC-03 へのリンクはメニューへ退避 | `QuickPost.tsx` / ホーム |
| S2 | 記帳後に「仕分けタグの確認・修正」をインライン表示（既存 autoCategory を編集可能に） | `QuickPost.tsx` ＋ タグ更新 API |
| S3 | 「＋数値も残す」タップ入力 UI（バイタル/睡眠/体重をチップ・ステッパー化）。既存の各カテゴリ作成 API を流用 | 新コンポーネント ＋ 既存 `/api/records` 系 |
| S4 | 過去日付（#30）・一括取り込み（#31）・音声（#34）の導線を折りたたみで追加 | 各 issue と連携 |

- API 側は**新規エンドポイント不要**（quick-entry ＋ 既存カテゴリ作成を再利用）。主に web の IA / コンポーネント変更。
- S1〜S2 を先行リリースし、テスターに再評価してもらってから S3 以降を詰める。

## 6. 受け入れ条件 (issue #33)

- [ ] ホームの入力入口が 1 つに統合され、迷いが生じない
- [ ] 即時記帳の即レス体験（#1）を維持
- [ ] 基礎データがタップ/選択で入力完結（自由記述の強制なし）
- [ ] 1 スクロールで「書く→返る→数値追加」が完結
- [ ] 65 歳ペルソナ「平岡みち子」＋ brain fog 配慮（大きいタップ領域・少ない手数）
- [ ] ライトテーマ固定（BR-09）・既存デザイン原則を踏襲

## 7. リスク / 論点

- **#1 の即時記帳思想との整合**: 「今を書く」体験は壊さない（むしろ強化）。山口さん要望（#1）と矛盾しないことを実装前に確認。
- **構造化データの記録率低下**: ②を撤去すると数値記録が減る懸念 → S3 のタップ UI で「むしろ増える」状態を目指し、計測（記録カテゴリ別件数）で検証。
- **#35（ライフログ志向）と一体で文言設計**: プレースホルダ/コピーは #35 のトーン&マナーと合わせて決める。
