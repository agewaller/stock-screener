# Interaction States — 共通インタラクション

データ取得・操作の **4 状態** (Loading / Empty / Error / Success) の表現規約。全画面で一貫させる。

## 4 状態の定義と表現

### Loading (読み込み中)

- **発生**: API 呼び出し中、ページ初期表示中、フォーム送信中、AI ストリーミング開始前
- **表現**:
  - ページ全体: スケルトン UI (骨組みを灰色のブロックで表示)
  - 部分: スピナー (小) + 操作対象を非活性化
  - フォーム送信: 送信ボタンをスピナー化、フォーム全体を read-only
  - AI ストリーミング: SSE chunk 到着前は「考えています…」、到着開始後は逐次表示
- **時間目安**:
  - 100ms 未満: 何も表示しない (瞬時に見える)
  - 100ms 〜 1s: スピナー
  - 1s 以上: スケルトン + 「読み込んでいます…」キャプション
- **アクセシビリティ**: `aria-busy="true"` を該当領域に付与

### Empty (データなし)

- **発生**: タイムラインが 0 件、過去記録が 0 件、ai_conversations が 0 件、PubMed 検索結果 0 件等
- **表現**:
  - アイコン or イラスト (中央配置、控えめ)
  - 「まだ記録がありません」「相談履歴がありません」等の状況説明
  - 復帰アクション (「今日の体調を記録する」ボタン等) を**必ず置く**
- **アクセシビリティ**: 見出し (`h2` 程度) として読み上げ可能に
- **ペルソナ配慮 (65 歳)**: 「使い方が分からない」状態を最初に経るので、Empty 状態は特に丁寧に。「まだ何もないのは普通のことです、最初の 1 件を試してみましょう」のような穏当なトーン

### Error (エラー)

- **発生**: API エラー (4xx/5xx)、バリデーションエラー、ネットワーク切断、AI 月次キャップ到達 (422)
- **表現**:
  - 軽微 (バリデーション): フォーム上部にメッセージブロック、該当項目に赤枠
  - 致命 (500 系・ネットワーク): 全画面エラーまたは大きめのバナー
  - **コストキャップ 422 (BR-20)**: 「今月の利用上限に達しました」モーダル、月次集計の表示と「翌月までお待ちください / プランを上げる相談」リンク
  - 復帰アクション (再試行・前に戻る) を**必ず置く**
- **エラーコード対応**: 詳細設計書 §3 を参照 (api-list.md にもエラーコード列挙あり)
- **トーン**: 利用者を責めない。「入力内容をご確認ください」「通信に失敗しました」など中立的に
- **オフライン時 (PII 禁止原則の帰結)**: 「インターネット接続が必要です。電波の良い場所でもう一度お試しください」(BR-15 と整合、書きかけの内容は破棄される旨も明示)
- **アクセシビリティ**: `role="alert"` + フォーカスをエラー領域に移す
- **PII 配慮 (BR-16/18)**: エラーメッセージにユーザー本文・PII を含めない (server log 側も同様、構造化ログ + ホワイトリスト)

### Success (成功)

- **発生**: 保存完了、削除完了、ログイン成功、エクスポート開始受付 (202)、AI 応答完了
- **表現**:
  - 一過性: **トースト** (画面右下 / モバイルは下部から 3 秒間表示、自動消失)
  - 重要: **モーダル**で明示的に閉じてもらう (退会完了 / エクスポート完了 等)
  - インライン: フォーム上部に成功メッセージ (編集の更新等)
- **トーン**: 簡潔・能動態。「保存しました」「ログアウトしました」「エクスポートを受け付けました」
- **アクセシビリティ**: `role="status"` で読み上げ

## フィードバック手段の使い分け

| 手段 | 用途 | 持続 |
|---|---|---|
| トースト | 軽微な成功・情報 | 3 秒で自動消失 |
| インラインメッセージ | フォーム送信結果・バリデーション | 操作で消えるまで |
| モーダル | 確認・致命的なエラー・退会のような重要な完了 | 利用者が閉じるまで |
| バナー | 全画面共通のお知らせ (メンテ予告 / 移行案内 FN-ADMIN-07 等) | 閉じるか期間で |

## 共通ルール

1. **状態の重複を避ける**: 1 領域で同時に複数の状態を表示しない (Loading 中の Empty 表示はしない)
2. **遷移の予告**: 削除や退会など破壊的操作の前には必ずモーダルで確認 (BR-08: `confirm()` 不使用、インライン UI で代替)
3. **戻れる動線**: Error 状態でも「前に戻る」「再試行」を必ず提供
4. **沈黙しない**: 1 秒以上待たせるなら必ず何らかの Loading 表示
5. **エラー文言**: 利用者にエラーコードを直接見せる場合は技術詳細は出さず、サポートに伝えるための短い ID (例: trace_id 末尾 8 桁) のみ
6. **AI 文言の言い換え (BR-09)**: Loading/Success/Error の文言に「AI」リテラルを混ぜない。「相談を準備しています」「ひとことを作成中」等

## 画面別の状態当てはめ例

| 画面 | Loading | Empty | Error | Success |
|---|---|---|---|---|
| SC-01 ランディング | (即時) | — | Identity Platform へ繋がらない (Firebase signIn 失敗): 「ログイン画面に接続できません」 + 再試行 | (リダイレクト) |
| SC-02 タイムライン | スケルトン (リスト領域) | 「まだ記録がありません」+ 「最初の記録を試す」ボタン | 全画面エラー + 再試行 | — (記録完了は SC-03 から戻った時にトースト) |
| SC-03 記録入力 | 送信ボタンスピナー、フォーム read-only | — | フォーム上部メッセージ + 該当項目に赤枠 | SC-02 に戻ってトースト「記録しました」 |
| SC-04 解析結果 | スケルトン + 「ふりかえりを作成中…」 | — (記録 0 件なら「記録を 3 日以上ためてみましょう」) | 422 キャップ: モーダル / それ以外: 全画面エラー | — |
| SC-05 会話 | 「考えています…」 → SSE chunk 流入 | 会話 0 件: 「最初の相談を始める」CTA | チャンク途中エラー: 「ここまでで止まりました」+ 「続きを試す」ボタン | (会話完了は無音、メッセージ末尾に時刻のみ) |
| SC-06 医師レポート | 「レポートを作成中… 30 〜 60 秒」 | 過去 0 件は説明 + CTA | 422 キャップ / 生成失敗ともインラインエラー | プレビュー表示 (PDF 化はトースト「保存しました」) |
| SC-07 履歴 | カレンダー/リストスケルトン | 「この期間の記録はありません」 | 全画面エラー | — |
| SC-08 PubMed | スケルトン (検索結果領域) | 「該当する論文が見つかりませんでした」 | API エラー (外部) は明示 | (要約成功はインライン) |
| SC-09 エクスポート | 受付直後はインライン「準備しています…」 | 過去 0 件は説明のみ | バリデーション (期間矛盾等) はフォーム上部 | トースト「エクスポートを受け付けました」+ 完了時メール |
| SC-10 監査ログ | スケルトン | 履歴 0 件 (初日のみ) は「これからアクセス履歴が記録されます」 | 全画面エラー | — |
| SC-11 設定 | プロファイル領域スケルトン | — | フォーム上部 | インライン「変更を保存しました」/ 退会完了はモーダル |
| SC-12 管理画面 | 各タブごとのスケルトン | サブ機能ごと (例: 通知キャンペーン 0 件) | フォーム or 全画面 | 一過性はトースト、重要 (鍵更新等) はインライン |
