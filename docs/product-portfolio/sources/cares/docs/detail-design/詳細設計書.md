## 1. 認証・認可の詳細設計

[ADR-0014](../adr/0014-auth-switch-to-identity-platform.md) に従い、認証は **GCP Identity Platform (Firebase Auth)** + **Auth.js v5 Credentials provider** で実装。

### 1.1 認証フロー (Firebase ID token → Auth.js session cookie)

詳細シーケンス: [`../architecture/sequence/auth-login.md`](../architecture/sequence/auth-login.md)

実装ポイント:

- **クライアント**: `apps/web/lib/firebase-client.ts` で Firebase JS SDK を初期化、`setPersistence(indexedDBLocalPersistence)` を強制 (ADR-0007 遵守、localStorage に refresh token を置かない)
- **ログイン**: `loginWithGoogle()` (popup) / `loginWithPassword(email, pw)` で Firebase ID token を取得
- **セッション**: ID token を Auth.js の Credentials provider に POST → サーバで `firebase-admin.auth().verifyIdToken()` → Auth.js が HttpOnly cookie (`authjs.session-token`) でセッション格納
- **セッション内容**: `accessToken` (= Firebase ID token、1h TTL), `provider` ("password"/"google.com"), `user.email`, `user.name`
- **セッション TTL**: Auth.js `maxAge: 60 * 60` (1h、Firebase ID token と整合)
- **API 認証**: BE は `Authorization: Bearer <ID token>` から `firebase-admin.auth().verifyIdToken()` で検証 (apps/api/src/index.ts の middleware)
- **長時間滞在対策**: 1h 経過で ID token が expire するため、API 呼び出し前に `getCurrentIdToken()` で Firebase SDK から fresh token を取り直す (SDK が refresh token で自動更新)

### 1.2 認可 Middleware

すべての BE エンドポイントを覆う共通 middleware (`apps/api/src/index.ts` 内、`app.use("/api/*", ...)`):

```
要求受信
  ↓
0. x-admin-token break-glass チェック (Tier 3)
  ↓ (該当なし)
1. Authorization: Bearer <token> から ID token 抽出
  ↓
2. firebase-admin.auth().verifyIdToken(idToken)
   (cert は Google が rotate、admin SDK が内部キャッシュ)
  ↓
3. decoded.uid を主軸として prisma.user.upsert({where:{globalUid: decoded.uid}})
   (初回ログインで自動 provision)
  ↓
4. c.var.userId / email / roles / isAdmin / provider に格納
  ↓
5. 管理 API なら追加で /api/admin/* gate (1.3 の三段チェック)
  ↓
6. 監査ログは各ハンドラ内で recordAudit() を fire-and-forget
  ↓
ハンドラ実行
```

### 1.3 管理者認可の三段フェイルセーフ (ADR-0011 改訂 / ADR-0014)

実装は以下の OR 判定 (`apps/api/src/index.ts` の middleware 内):

```typescript
// Tier 3: break-glass (Firebase 全停止時の救済、middleware の最先頭で判定)
const breakGlass = c.req.header("x-admin-token");
if (breakGlass && ADMIN_BREAKGLASS_TOKEN && breakGlass === ADMIN_BREAKGLASS_TOKEN) {
  // OWNER_EMAIL の user を upsert + isAdmin=true で通す + 警告ログ
  console.warn(JSON.stringify({ event: "breakglass_admin_used", path: c.req.path }));
  return next();
}

// Firebase verify 成功後
const decoded = await adminAuth.verifyIdToken(token, false);
const provider = decoded.firebase.sign_in_provider; // "password" | "google.com" | ...

// Tier 1: Firebase custom claim
const claimAdmin = decoded.admin === true;

// Tier 2: OWNER_EMAIL + password provider (Google 乗っ取り対策)
const isOwner = decoded.email?.toLowerCase() === OWNER_EMAIL.toLowerCase();
const ownerByPassword = isOwner && provider === "password";

const isAdmin = claimAdmin || ownerByPassword;
```

- **Tier 2 の核心**: OWNER (`agewaller@gmail.com`) は **email/password でログインしたときだけ** admin になる。Google ログインでは一般ユーザ扱い (Gmail アカウント乗っ取り時の管理画面アクセス遮断)
- **break-glass token** は `ADMIN_BREAKGLASS_TOKEN` env (`openssl rand -hex 32` で生成、`.env` で配布、本番は Secret Manager)、`x-admin-token` ヘッダ一致で Firebase verify をスキップして通す。使用時は必ず `breakglass_admin_used` 警告ログを出す
- **custom claim 付与**: `apps/api/scripts/grant-admin.ts <email>` で `setCustomUserClaims(uid, {admin:true})` + `revokeRefreshTokens()` (強制再ログインで新 token に claim を載せる)

### 1.4 セッション破棄

`POST /api/auth/signout` (Auth.js v5 が自動提供):

1. Auth.js の cookie (`authjs.session-token`) を max-age=0 で削除
2. `apps/web/components/Header.tsx` で先に `clientSignOut()` (= Firebase `fbSignOut`) を呼んで IndexedDB の `firebaseLocalStorageDb` をクリア
3. ブラウザ側のメモリ上 blob URL は unmount で `URL.revokeObjectURL` (PhotoCard 等)

> **旧 Keycloak 設計の Backchannel logout は廃止 (ADR-0014)**。cares は単独サービスの認証セッションのみ破棄する。ValuationMatrix / zen-track との SSO 連携は将来別 ADR (Identity Platform を共有する形で再設計予定)。

### 1.5 IndexedDB 暗号化キャッシュ鍵の払い出し

`POST /api/cache-key`:

```typescript
app.post('/api/cache-key', authMiddleware, async (c) => {
  const user = c.get('user')

  // ランダム 256bit 鍵を生成
  const keyBuf = crypto.getRandomValues(new Uint8Array(32))
  const keyB64 = Buffer.from(keyBuf).toString('base64')

  // 監査ログ: 鍵払い出し
  await auditLog(c, {
    action: 'write',
    targetTable: 'session_cache_keys',
    details: { event: 'cache_key_issued' },
    visibleToUser: true,
  })

  // 鍵は DB には永続化しない（クライアントだけが持つ）
  return c.json({
    key: keyB64,
    validUntil: new Date(Date.now() + 24 * 60 * 60 * 1000).toISOString(),
  })
})
```

- 鍵は **クライアントだけが持つ**（ログアウトで完全消失）
- サーバが鍵を保持しないため、サーバ侵害時もキャッシュは復号不可能
- ただしユーザーがアクティブセッション中なら、トークンと一緒に鍵もメモリにある（passive read 防御のみ）

## 2. 暗号化実装の詳細

> **実装状況 (2026-06-04 更新)**: フィールド暗号化基盤を実装（issue #13）。
>
> - 鍵: `DATA_ENCRYPTION_KEY`（256bit / 64 hex）。local/staging は env、prod は Secret Manager 注入。
> - 実装: `packages/db/src/encryption.ts`（AES-256-GCM）+ `packages/db/src/index.ts` の Prisma Client
>   Extension（`createPrismaClient()`）で対象カラムを透過的に encrypt/decrypt。
> - **格納形式**: 列型変更（Bytes）を避け migration リスクを抑えるため、既存の `String` 列に
>   バージョン付きエンベロープ `enc:v1:` + base64(nonce12 | tag16 | ct) で格納する。
>   プレフィックス無しの旧平文は read で素通し（移行安全）。`key_version` 相当は `v1` タグで表現。
> - **適用済**: `text_entries.content`。
> - **未適用（テーブル未作成。各機能フェーズで作成時に適用）**: `ai_messages.content`(#2),
>   `drafts.payload`(#18), `oauth_tokens.encrypted_refresh_token`(#4), `user_profiles.*`。
> - 既存平文の暗号化移行: `apps/api/scripts/encrypt-backfill.ts`（idempotent / `--apply`）。
> - prod の本番データ暗号化はフェーズ 7（データ移行）時に本番鍵で実施。

### 2.1 サーバ側暗号化（DB カラム）

対象: `text_entries.content`, `ai_messages.content`, `user_profiles.current_medications`, `user_profiles.allergies`, `oauth_tokens.encrypted_refresh_token` 等

#### 鍵管理

- マスター鍵を Secret Manager の `DATA_ENCRYPTION_KEY`（256bit）に保管
- 起動時に BE が取得、メモリで保持
- ローテーション: 半年に 1 回、新旧鍵の二重デコード期間あり（`key_version` 列を持たせる）

#### 暗号方式

- **AES-256-GCM**
- nonce は 12 バイトランダム生成、暗号文と共に格納
- 認証タグも暗号文に含める

#### Prisma 統合

`packages/db/encryption.ts`:

```typescript
export function encrypt(plaintext: string): Buffer {
  const nonce = crypto.randomBytes(12)
  const cipher = crypto.createCipheriv('aes-256-gcm', masterKey, nonce)
  const enc = Buffer.concat([cipher.update(plaintext, 'utf-8'), cipher.final()])
  const tag = cipher.getAuthTag()
  return Buffer.concat([nonce, tag, enc])  // 12 + 16 + N
}

export function decrypt(blob: Buffer): string {
  const nonce = blob.subarray(0, 12)
  const tag = blob.subarray(12, 28)
  const enc = blob.subarray(28)
  const decipher = crypto.createDecipheriv('aes-256-gcm', masterKey, nonce)
  decipher.setAuthTag(tag)
  return Buffer.concat([decipher.update(enc), decipher.final()]).toString('utf-8')
}
```

Prisma Middleware で `text_entries`, `ai_messages` 等の特定テーブルの特定カラムに対して自動 encrypt/decrypt。

### 2.2 クライアント側暗号化（IndexedDB）

`apps/web/lib/encryptedIndexedDB.ts`:

```typescript
export async function setEncrypted(key: string, value: unknown) {
  const sessionKey = await getSessionKey()  // メモリから
  const cryptoKey = await crypto.subtle.importKey(
    'raw', sessionKey, { name: 'AES-GCM' }, false, ['encrypt']
  )
  const nonce = crypto.getRandomValues(new Uint8Array(12))
  const enc = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv: nonce }, cryptoKey, encoder.encode(JSON.stringify(value))
  )
  const blob = new Uint8Array(nonce.length + enc.byteLength)
  blob.set(nonce); blob.set(new Uint8Array(enc), nonce.length)
  const db = await openDB()
  await db.put('cache', blob, key)
}
```

- 鍵はメモリ + IndexedDB メタ（`indexedDB-meta`）に保存。タブ間共有不要
- メタには `validUntil`, `userIdHash` も保存し、別ユーザーが同タブで使った場合の混線を防ぐ

## 3. 監査ログ実装の詳細

### 3.1 ログ書き込み

全 BE エンドポイントを覆う middleware:

```typescript
app.use('*', async (c, next) => {
  const startTs = Date.now()
  await next()
  const action = inferAction(c.req)  // read/write/delete/export
  const target = inferTarget(c.req, c.res)
  await db.auditLog.create({
    data: {
      eventAt: new Date(),
      actorType: c.get('user')?.isAdmin ? 'admin' : 'self',
      actorUserId: c.get('user')?.id,
      action,
      targetTable: target.table,
      targetId: target.id,
      targetUserId: target.userId,
      details: { method: c.req.method, path: c.req.path, durationMs: Date.now() - startTs },
      visibleToUser: target.userId === c.get('user')?.id || c.get('user')?.isAdmin,
    },
  })
})
```

### 3.2 ユーザー可視化

`GET /api/audit-logs/me`:

```sql
SELECT id, event_at, actor_type, action, target_table,
       LEFT(target_id::text, 6) AS target_id_short, details
FROM audit_logs
WHERE target_user_id = $1
  AND visible_to_user = TRUE
ORDER BY event_at DESC
LIMIT 50;
```

- 具体 `actor_user_id` は返さず、`actor_type` のみ（プライバシー）
- `target_id` は短縮（識別感は残しつつ詳細は隠す）

### 3.3 ログ自体のセキュリティ

- 監査ログ自体の改ざんを防ぐため、append-only（UPDATE/DELETE は管理者でも不可）
- Cloud SQL の権限分離: BE のサービスアカウントは INSERT のみ可、別の admin SA に DELETE 権を限定

```typescript
const cost = (inputTokens * pricing.inputUSDPer1M + outputTokens * pricing.outputUSDPer1M)
  / 1_000_000 * USD_TO_JPY_RATE
await db.aiUsageLog.create({
  data: { userId, provider, model, inputTokens, outputTokens, costJpy: cost, purpose: 'chat' }
```

### 4.2 キャップ判定

`GET /api/ai/quota`:

```sql
SELECT COALESCE(SUM(cost_jpy), 0) AS used
FROM ai_usage_logs
WHERE user_id = $1
  AND date_trunc('month', created_at) = date_trunc('month', NOW());
```

`UserQuota` テーブル（または `user_preferences.feature_flags`）に `monthly_cap_jpy` を保持。デフォルト 1000 円、管理者が個別上書き可能。

### 4.3 キャップ enforcement

middleware:

```typescript
async function quotaMiddleware(c, next) {
  if (c.req.path.startsWith('/api/ai/')) {
    const { used, cap } = await getQuota(c.get('user').id)
    if (used >= cap) {
      return c.json({ error: { code: 'QUOTA_EXCEEDED', message: '今月の上限に達しました' } }, 422)
    }
  }
  await next()
}
```

縮退モデルへのフォールバック（Opus → Haiku）は MVP では実装せず、エラー返却のみ。

## 5. エラーハンドリング戦略

### 5.1 階層

```
Hono app.onError
  ↓ 既知のエラークラスに対応した HTTP ステータス
  ↓ クライアントには { error: {code, message, details}, requestId } 形式で返す
  ↓ Cloud Logging に詳細スタックを記録
```

### 5.2 エラークラス

`packages/shared/errors.ts`:

```typescript
export class ValidationError extends BaseError { code = 'VALIDATION_ERROR'; status = 400 }
export class UnauthorizedError extends BaseError { code = 'UNAUTHORIZED'; status = 401 }
export class ForbiddenError extends BaseError { code = 'FORBIDDEN'; status = 403 }
export class NotFoundError extends BaseError { code = 'NOT_FOUND'; status = 404 }
export class ConflictError extends BaseError { code = 'CONFLICT'; status = 409 }
export class QuotaExceededError extends BaseError { code = 'QUOTA_EXCEEDED'; status = 422 }
export class RateLimitedError extends BaseError { code = 'RATE_LIMITED'; status = 429 }
```

### 5.3 ユーザー向けメッセージ

- すべて日本語、専門用語を避ける
- 「AI」リテラルは使わない（旧仕様継続）
- 例: `QUOTA_EXCEEDED` → 「今月の利用量が上限に達しました。来月になると再びご利用いただけます。」

### 5.4 AI Provider のエラー

- 503 / 529（Anthropic 過負荷）: ユーザーには「混雑しています。しばらく後にお試しください」
- タイムアウト（30 秒超過）: 同上
- 自動リトライ・自動フォールバックは行わない（ADR-0006、複雑化禁止）

### 5.5 コストキャップ超過時

- `QUOTA_EXCEEDED` (422) を返す
- ユーザーメッセージ: 「今月のご利用上限に達しました。来月 1 日にリセットされます。」
- **縮退モデルへの自動切替は行わない**（CLAUDE.md フォールバックチェーン禁止）
- 管理者がユーザー個別キャップを上書きすれば再開可能

## 6. Identity Platform セットアップ

[ADR-0014](../adr/0014-auth-switch-to-identity-platform.md) 採用に伴い、認証は GCP Identity Platform (Firebase Auth) を使う。Keycloak Realm 設計は廃止。

### 6.1 GCP プロジェクト

- 現在の dev/prod 共用プロジェクト: `arctic-anvil-497002-q2` (BMP cares-advisers-jp Project)
- 将来 dev/prod を分離する場合は `cares-prod` / `cares-staging` / `cares-dev` の 3 プロジェクトに分割し、それぞれで Identity Platform を独立して有効化

### 6.2 Provider 設定 (Identity Platform Console / REST API)

| Provider | 状態 | 備考 |
|---|---|---|
| **Google** | ✅ 有効 | client ID / secret は Firebase が auto-provision (Console から「有効にする」操作のみで完了) |
| **Email / Password** | ✅ 有効 | パスワードリセットは Firebase 標準の reset email を使う |
| 匿名ログイン | ⚠ 設計のみ、MVP では無効 | FN-AUTH-03、将来必要なら有効化 |
| Magic Link | ⚠ 設計のみ | Firebase の sign-in with email link を使う、現状未使用 |

### 6.3 ロール / Custom Claim

Keycloak Realm のロールは廃止。代わりに **Firebase custom claim** を使う。

| Claim | 値 | 説明 |
|---|---|---|
| `admin` | `true` (オプション) | 管理者。`apps/api/scripts/grant-admin.ts <email>` で付与、`revokeRefreshTokens` で強制再ログイン |

複数管理者をサポート。`support` ロールは廃止 (必要なら将来 `support: true` を追加)。

### 6.4 認証経路

| 経路 | クライアント | サーバ |
|---|---|---|
| Google ログイン | `signInWithPopup(GoogleAuthProvider)` → ID token | `firebase-admin.verifyIdToken()` → custom claim を見て isAdmin 判定 |
| Email/Password ログイン | `signInWithEmailAndPassword()` → ID token | 同上、`provider==="password"` のとき OWNER は Tier 2 で admin 昇格 |
| パスワードリセット | `sendPasswordResetEmail(email)` | Firebase が自動でメール送信 → reset ページに飛ぶ |

### 6.5 Authorized Domains

- `localhost` (dev)
- `cares.advisers.jp` (prod)
- `arctic-anvil-497002-q2.firebaseapp.com` (Identity Platform default、メール経由のリセットページ用)
- `arctic-anvil-497002-q2.web.app` (同上、予備)

REST API で更新: `PATCH /v2/projects/{project}/config?updateMask=authorizedDomains`

### 6.6 Service Account

- 名前: `cares-firebase-admin@arctic-anvil-497002-q2.iam.gserviceaccount.com`
- ロール: `firebaseauth.admin`, `identitytoolkit.admin`, `storage.objectAdmin` (写真機能用), `iam.serviceAccountTokenCreator` (v4 Signed URL 用)
- 鍵: `infra/firebase-sa.json` (gitignored、chmod 600)、`.env` の `FIREBASE_SERVICE_ACCOUNT_JSON` に 1 行 JSON で渡す
- 本番: Secret Manager に格納 → Cloud Run のサービス ID で読み出す

### 6.7 オーナー初期化手順 (1 回だけ)

```sh

# 2. 上のリンクからブラウザで初期パスワード設定
```

### 6.8 cares 単独 / 複数サービス SSO

現在は cares 単独で Identity Platform を使う。ValuationMatrix / zen-track との共通アカウント運用は、同じ Identity Platform プロジェクトを共有する形で将来再設計 (別 ADR)。

> 旧 Keycloak Backchannel logout は廃止。cares は単独サービスの認証セッションのみ破棄する (詳細は §1.4)。

## 7. テスト戦略

> **実装状況 (2026-06-01)**: 単体 (`unit-node` / `unit-web`) + 統合 (`integration`) + E2E (Playwright) は実装済（`pnpm test` / `pnpm test:e2e`）。

### 7.1 テストピラミッド

| 層 | 対象 | フレームワーク | 比率 |
|---|---|---|---|
| 単体 | Pure 関数、Zod スキーマ、暗号化、認可ロジック | Vitest | 70% |
| 統合 | BE エンドポイント、DB 含む | Vitest + 実 Postgres (`cares_test` DB) | 25% |
| E2E | ブラウザ操作のシナリオ | Playwright | 5% |

### 7.2 単体テスト

- `packages/shared/*` は完全な型カバレッジ + 全ロジックのテスト
- 暗号化・復号は roundtrip テストを必須
- 認可 middleware の三段フェイルセーフ各パスをテスト

### 7.3 統合テスト

- 各 BE エンドポイントについて、Postgres 起動 + Prisma migrate + ハンドラ呼び出し
- 認可エラーケース、バリデーションエラーケースもカバー
- AI 呼び出しはモック（Anthropic SDK をスタブ）

### 7.4 E2E テスト

- 重要シナリオ: ログイン → 症状記録 → AI 解析 → エクスポート
- Playwright で実 Identity Platform (Firebase Auth) を使ってログインフローも検証（Emulator 不採用、ADR-0014。テスト用ユーザの準備は `e2e/README.md` 参照）
- E2E は実 Firebase 認証情報とフルスタックが必要なため **CI には含めない**（ローカル / 手動実行）
- staging 環境での毎日の smoke test は**未実装**（staging 環境は [ADR-0015](../adr/0015-staging-environment-in-prod-project.md) で構築済 (2026-06-02) だが、自動 smoke は GitHub Actions CI/CD と併せてフェーズ 7 以降）

### 7.5 性能テスト（将来計画・未実装）

- k6 で API レイテンシを測定
- 想定負荷: ユーザー 50 人同時、5 req/min/user
- p95 < 500ms を目標

### 7.6 セキュリティテスト（将来計画・未実装）

- ZAP で OWASP Top 10 自動スキャン（CI で staging に対して）
- 認可ガード bypass テスト（他人の ID を URL に入れる等）→ 一部は統合テストでカバー済

## 9. データ移行スクリプト設計

### 9.1 配置

`apps/migration/`：

```
apps/migration/
├─ src/
│  ├─ index.ts            # エントリ
│  ├─ users.ts            # 旧 Firebase Auth → Identity Platform (同 Firebase 基盤)
│  ├─ firestore-to-pg.ts  # Firestore → Postgres
│  ├─ storage-copy.ts     # Firebase Storage → Cloud Storage
│  └─ verify.ts           # 差分確認
└─ Dockerfile
```

Cloud Run Job として実行。

### 9.2 旧 Firebase Auth → Identity Platform (Firebase Auth)

> **ADR-0014**: 新旧とも Firebase Auth 基盤のため、旧設計（Keycloak への移行）と比べ認証移行は大幅に単純化される。詳細は要再設計（ADR-0010 改訂注記参照）。

```typescript
// 旧・新ともに Firebase Auth 基盤のため (ADR-0014):
// - 同一 Identity Platform プロジェクトに集約する場合: UID をそのまま再利用でき、認証移行は実質不要
// - 別プロジェクト間移行の場合: Firebase Admin SDK の importUsers で
//   パスワードハッシュ込みエクスポート/インポート (Google プロバイダは federated identity を維持)
// - Anonymous: FN-AUTH-03 未実装のため移行対象なし
// - users.global_uid に Firebase UID を直保管 (user_link は廃止)
```

### 9.3 Firestore → Postgres

各コレクションを順に処理:

```typescript
async function migrateSymptoms(uid: string, userId: string) {
  const snap = await firestore.collection(`users/${uid}/symptoms`).get()
  for (const doc of snap.docs) {
    const data = doc.data()
    await prisma.symptom.create({
      data: {
        id: data.id,  // 既存 ID 保持
        tenantId: userId,
        userId,
        recordedAt: data.createdAt.toDate(),
        conditionLevel: data.condition_level,
        sleepQuality: data.sleep_quality,
        memo: data.memo,
      },
    })
  }
}
```

- 暗号化対象カラム（text_entries, ai_messages）は移行時に暗号化
- 写真は Firebase Storage から Cloud Storage に `gsutil rsync`
- 全件処理後 `apps/migration/src/verify.ts` で count 一致を確認

### 9.4 並列度・性能

- ユーザー単位で並列化（10 並列程度）
- 大きなコレクションはバッチ化
- 失敗時はリトライ、最終的にユーザー別エラーログ

### 9.5 ロールバック

- 移行前に Firestore のスナップショットを Cloud Storage にダンプ
- 失敗時は DNS を旧 cares に戻す + Postgres は drop + 再試行
